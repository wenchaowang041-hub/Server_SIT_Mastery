# 2026-03-29 在家补习: 整机 SIT 起手式训练

## 今日补习目标

- 训练你在没有现场问题时也能持续提升的方法论能力
- 固化“拿到一台服务器后的前30分钟动作”
- 为鲲鹏 / 昇腾 / AI 服务器整机 SIT 建立统一检查框架

## 先记住一句话

整机 SIT 不是上来就压测，而是先判断这台机器值不值得开始压测。

如果基线没建好，后面的 CPU、内存、网络、NPU、整机稳定性结论都可能失真。

## 一、拿到一台服务器后的标准顺序

### 第1步：确认机器身份

你要先回答这台机器是谁，而不是它快不快。

重点确认：

- 型号
- 序列号
- 主板信息
- BIOS 版本
- BMC 版本
- CPU 型号和路数
- DIMM 数量和容量

常用命令：

```bash
hostnamectl
dmidecode -t system
dmidecode -t bios
dmidecode -t baseboard
dmidecode -t processor
dmidecode -t memory
ipmitool mc info
ipmitool fru
```

你要形成的判断习惯：

- 这是不是我要测的那台机器
- 它是不是正确版本
- 它有没有被换过板卡、CPU、DIMM、SSD

### 第2步：确认资源有没有被系统完整识别

重点确认：

- CPU 逻辑核数是否正确
- NUMA 节点是否正常
- 内存总量是否正确
- NVMe / RAID / 网卡 / GPU / NPU / DPU 是否全部枚举

常用命令：

```bash
lscpu
lscpu -e=cpu,node,socket,core,online
numactl -H
free -h
lsblk
lspci
lspci -nn
```

你要形成的判断习惯：

- 少一路 CPU，不一定是 OS 问题，可能是 BIOS、硬件、插槽、故障 CPU
- 少内存，不一定是系统问题，可能是 DIMM 插法、坏条、降频、屏蔽
- 少设备，不一定是驱动问题，先看 PCIe 枚举

### 第3步：确认拓扑是不是健康

整机 SIT 不只是“有没有”，更重要的是“挂得对不对”。

重点确认：

- NUMA 是否均衡
- PCIe 链路是否降速降宽
- 高速网卡、GPU、NPU 是否挂在合理的 root complex 下

常用命令：

```bash
numactl -H
lspci -vv
```

重点看什么：

- `LnkCap`
- `LnkSta`
- socket/node/core 分布
- 关键设备是否全部 online

典型风险：

- 设备在，但链路只跑到预期的一半
- NUMA 不均衡，导致网络压测和 AI 负载结果波动很大

### 第4步：确认健康度

现场最常见的问题，不是“测不出性能”，而是机器本来就不健康。

重点确认：

- 温度
- 风扇
- 电压
- 电源状态
- SEL 历史告警

常用命令：

```bash
ipmitool sensor
ipmitool sdr list full
ipmitool sel list
```

你要形成的判断习惯：

- 有持续过温，不要急着跑烧机
- 有 PSU 抖动或掉电记录，不要急着下整机稳定性结论
- 有风扇异常，不要直接开始 AI 训练负载

### 第5步：确认软件栈版本

硬件工程师如果不看软件栈，就很容易把版本问题误判成硬件问题。

重点确认：

- OS 版本
- kernel 版本
- 驱动版本
- 固件版本
- 关键工具链版本

常用命令：

```bash
cat /etc/os-release
uname -a
lsmod
ethtool -i eth0
modinfo mlx5_core
modinfo nvme
```

如果是 AI 服务器，还要补：

```bash
nvidia-smi
npu-smi info
```

## 二、什么情况下不能直接进入专项测试

下面这些情况，一旦出现，要先拦截，不要急着继续：

- 实际配置和送测配置不一致
- BIOS/BMC 版本不在项目支持范围
- CPU、DIMM、SSD、NIC、GPU、NPU 枚举不完整
- PCIe 出现明显降速降宽
- SEL 里存在持续性过温、掉电、风扇异常
- NUMA 分布明显异常

这类场景下，正确动作不是“先测再说”，而是：

1. 记录现象
2. 保留原始日志
3. 给出风险判断
4. 先校准环境，再继续测试

## 三、今天你要背下来的判断框架

遇到任何机器，先按这5个问题过一遍：

1. 这台机器是谁
2. 它有什么资源
3. 这些资源挂得是否正确
4. 它现在健不健康
5. 它是否具备进入下一阶段测试的前提

如果这5个问题答不清楚，就不要急着做后面的性能、稳定性、兼容性结论。

## 四、面试式自测题

### 题1

现象：`lspci` 能看到网卡，但压测吞吐明显低于预期。  
你第一反应应该查什么？

标准思路：

- 查 `lspci -vv`
- 看 `LnkCap` 和 `LnkSta`
- 看是否存在 PCIe 降宽降速
- 再结合 NUMA 绑定和驱动版本判断

### 题2

现象：AI 训练任务一跑就掉速，机器没有直接报错。  
你最先怀疑什么？

标准思路：

- 查 BMC 传感器
- 查 SEL 是否有过温
- 查风扇策略和功耗墙
- 查 GPU/NPU 是否处于降频状态

### 题3

现象：CPU 核数对，内存总量也对，但业务性能抖动很大。  
你应该补看什么？

标准思路：

- 查 `numactl -H`
- 查 DIMM 分布是否均衡
- 查 NUMA 节点容量是否异常
- 查是否存在绑定策略错误
