## 一、整机系统图怎么理解

Day20 的整机系统图，建议按下面这 5 层来画：

```text
CPU / NUMA
-> PCIe
-> 关键设备
-> OS / Driver
-> BMC / 日志 / 工具
```

### 1. CPU / NUMA 层

这一层要放的信息：

- CPU 架构与型号
- Socket / Core / Thread / 逻辑 CPU 数
- NUMA 节点数
- 每个 NUMA 节点的 CPU 范围
- 每个 NUMA 节点的内存大小、可用内存和 distance

这一层解决的是两个问题：

- 这台机器的计算资源是什么结构
- 这些资源彼此之间访问代价如何

核心结论：

```text
总 CPU 和总内存说明资源上限，NUMA 决定路径成本。
```

### 2. PCIe 层

这一层要放的信息：

- PCIe 关键设备清单
- PCIe 枚举状态
- PCIe 拓扑
- 链路速率
- 链路宽度

这一层解决的是三个问题：

- 关键设备有没有全部在位
- 设备是通过什么路径接进系统的
- 链路是否存在降速、降宽

核心结论：

```text
设备是否能正常工作，首先取决于 PCIe 层是否正确枚举和正确建立链路。
```

### 3. 关键设备层

这一层重点关注：

- NVMe
- NIC
- NPU
- RAID
- HBA
- GPU（如有）

这些设备都走 PCIe 通道，是整机里最关键的高速设备。

这一层解决的问题是：

- 机器具备什么样的存储能力
- 机器具备什么样的网络能力
- 机器具备什么样的 AI 加速能力

### 4. OS / Driver 层

这一层必须单独画出来，因为：

```text
设备在位，不等于设备可用。
```

这层要看：

- OS 是否识别设备
- 驱动是否正常加载
- 初始化是否成功
- 日志中是否有明显错误

这一层解决的是：

- 设备是不是已经真正进入可用状态

### 5. BMC / 日志 / 工具层

这一层是观察层和验证层。

它不是服务器底层结构本身，而是工程师取证和验证的入口。

这里要看的内容包括：

- BMC 页面
- 传感器状态
- SEL
- `dmesg`
- `journalctl`
- 测试工具结果

这一层解决的是：

- 底层异常有没有证据
- 测试结果能不能被日志和状态支撑
- 当前问题应该从哪一层继续往下追

核心结论：

```text
工具提供结果，日志提供证据，BMC 提供带外视角。
```

## 二、NUMA 图怎么理解

NUMA 图最少应包含这 4 个信息：

- NUMA 节点数
- 每个节点的 CPU 范围
- 每个节点的内存分布
- 节点之间的 distance

NUMA 图的作用不是单纯展示参数，而是让你一眼看懂：

- 哪些 CPU 属于哪个 node
- 哪些 node 之间更近
- 哪些 node 之间更远
- 后面设备、IRQ、业务线程是否存在跨 NUMA、跨 socket 问题

NUMA 图的核心价值：

```text
它是后续理解设备路径、性能调优和资源绑定的底图。
```

## 三、PCIe 图怎么理解

PCIe 图最少应包含这 4 类信息：

- 关键设备
- PCIe 拓扑路径
- 枚举状态
- 链路状态

也就是要回答：

- 哪些设备挂在 PCIe 上
- 这些设备通过什么路径接上来
- 设备是否全部在位
- 链路速率和宽度是否正常

PCIe 图的核心价值：

```text
把“设备存在”这件事，落到可验证的链路和拓扑上。
```

## 四、存储图怎么理解

存储图最少应包含这 4 类信息：

- 存储设备类型
- 设备形态与容量
- 存储路径
- RAID / 逻辑盘关系

也就是要区分：

- HDD / SSD
- SATA / SAS / NVMe
- U.2 / U.3 / M.2
- 直通路径还是 RAID 路径

存储图的核心不是“盘有多少”，而是：

```text
物理盘是怎么一路变成 OS 下可见块设备的。
```

## 五、前20天最重要的10条结论

1. CPU 逻辑总数和内存总量只能说明服务器的资源上限，NUMA 和 PCIe 才决定真实的数据路径成本。
2. `lspci` 看到设备，只能说明 PCIe 枚举层已经识别到设备，不代表设备功能已经完全正常。
3. 设备是否真正可用，还必须继续结合 OS、驱动、日志和测试结果一起判断。
4. 测试不能只看结果，还要结合日志、设计文档和现象做证据分析，再决定是否提交 bug。
5. 遇到设备不识别或功能异常时，要按工具层、OS/驱动层、设备层、PCIe 层、物理连接层分层排查。
6. 做测试前要先冻结配置和日志基线，测试后第一时间收集日志并保持环境一致，避免问题复现失真。
7. 对简单重复的检查动作，应该沉淀成 SOP 或自动化脚本，减少时间浪费和人为手法误差。
8. BMC、SEL、`dmesg`、`journalctl` 是整机测试中的关键证据源，不能只盯着测试工具输出。
9. 存储、网络、NPU 这些关键设备的问题，最终都要回到 CPU、NUMA、PCIe 和驱动这条底层链路上理解。
10. 前20天最重要的收获不是多会几个命令，而是开始具备“看到服务器能画出结构、分层定位问题”的基础能力。
