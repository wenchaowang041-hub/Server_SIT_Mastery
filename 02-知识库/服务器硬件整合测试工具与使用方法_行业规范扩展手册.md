**1. 文档定位与行业测试思路**

服务器硬件整合测试的核心不是“把命令跑一遍”，而是用一套可复现、可追溯、可比对的方法验证平台是否达到交付、上线或批量部署要求。行业里常见的交付问题，往往不是设备完全不识别，而是拓扑不合理、固件版本不匹配、压测口径不统一、功耗与散热边界未被验证、日志未归档导致问题无法复现。

因此，一个成熟的测试文档至少要回答四个问题：测什么、用什么工具测、如何判断结果有效、出现异常后如何收敛定位。

| **测试层级** | **目标** | **典型输出** | **代表工具** |
|:--:|:--:|:---|:---|
| 环境基线 | 锁定可比条件 | OS/BIOS/固件/驱动/温度/功耗基线 | dmidecode, ipmitool, uname, lscpu |
| 硬件识别 | 确认设备完整可见 | CPU/内存/PCIe/磁盘/网卡/NPU清单 | lspci, lsblk, lsscsi, nvme |
| 拓扑校验 | 确认 NUMA 与 PCIe 归属合理 | NUMA 距离、链路速率、槽位映射 | numactl, hwloc, lspci -vv |
| 功能验证 | 确认组件可工作 | 链路 up、RAID 正常、温度正常、日志无错 | ethtool, ipmitool, storcli64 |
| 性能基线 | 建立吞吐/IOPS/延迟/带宽基线 | CPU/内存/存储/网络/NPU 跑分 | stress-ng, STREAM, fio, iperf3, ascend-dmi |
| 稳定性与边界 | 确认长稳与热边界 | 24h/72h 日志、峰值温度、峰值功耗 | stress-ng, ipmitool, dmesg, journalctl |
| 故障定位 | 快速归因并复盘 | 报错关键词、异常时间线、证据链 | dmesg, SEL, smartctl, journalctl |

| **行业上更看重的不是“跑分高”** 而是同一口径下结果可重复、负载定义清晰、测试前置条件受控、日志和配置可回放。这也是为什么存储测试会强调预处理与稳态、网络测试会强调流数与双向、功耗测试会强调负载点与环境温度。 |

**1.1 工具选型原则**

- 优先使用通用工具做“第一性确认”：比如 lspci、dmidecode、lsblk、ipmitool、dmesg，这些工具跨平台、跨厂商，适合先确认现象。

- 遇到控制器、BMC、加速卡等专有层问题，再转向厂商工具：如 storcli64、hiraidadm、PERCCLI、SSACLI、OneCLI、npu-smi。

- 性能测试工具必须与监控工具成对出现：fio 对 iostat，iperf3 对 ethtool/interrupts，stress-ng 对 mpstat/ipmitool，才能判断数据是否可信。

- 行业规范强调“原始证据保留”：压测日志、系统日志、固件版本、配置快照和异常时间点必须留档。

**2. 工具总览矩阵**

| **模块** | **首选工具** | **厂商/扩展工具** | **核心指标** | **使用阶段** |
|:--:|:---|:---|:---|:--:|
| 硬件识别 | lscpu, lspci, dmidecode, lsblk, lsscsi | hwloc/lstopo, nvme-cli | 数量、型号、槽位、链路、NUMA | 上机第一步 |
| CPU | stress-ng, mpstat, top | x86 可选 turbostat | 利用率、频率、温度、降频 | 功能 + 压测 |
| 内存 | memtester, STREAM, free | sysbench, EDAC/ECC 监控 | 容量、频率、带宽、ECC 错误 | 功能 + 基线 |
| 存储 | fio, iostat, smartctl, nvme-cli | mdadm, storcli64, hiraidadm, PERCCLI, SSACLI | BW、IOPS、lat、SMART、阵列状态 | 性能 + 故障 |
| 网络 | iperf3, ethtool, ping, mtr | sar -n, nstat, irqbalance | Gbps、Retr、丢包、协商速率、错误包 | 功能 + 性能 |
| BMC/OOB | ipmitool | Redfish, racadm, ilorest, OneCLI, SUM | 温度、风扇、功耗、SEL、远程控制 | 全流程 |

| **建议记忆方法** 不要按“命令名字”记，而要按“问题类型”记：识别问题看 lspci/lsblk，性能问题看 fio/iperf3/stress-ng，异常问题看 dmesg/SEL/smartctl，控制器问题看厂商 CLI。 |

**3. 硬件识别与拓扑工具**

上机后的前 10 到 15 分钟，决定了后续调试效率。行业里很少一上来就跑压力，通常先完成“识别、拓扑、版本、链路”四类确认。

**3.1 lscpu / numactl / hwloc-ls**

lscpu 用来确认 CPU 架构、逻辑核数、Socket、核心数和 NUMA 节点；numactl --hardware 用来确认 NUMA 距离和节点资源；hwloc 的 lstopo/hwloc-ls 适合把复杂服务器的 CPU、缓存、NUMA、PCIe 层级可视化，特别适合多 Socket、多加速卡平台。

**4. CPU 与内存测试工具**

**4.1 stress-ng：CPU/内存/系统稳定性压测主力**

stress-ng 是综合压力工具，适合做 CPU、内存、缓存、虚拟内存、文件系统与系统调用压力。它的价值不只是“把 CPU 跑满”，更在于能通过不同 stressor 触发不同路径，从而暴露降频、温升、内核异常和稳定性边界。

<table>
<colgroup>
<col style="width: 100%" />
</colgroup>
<thead>
<tr>
<th># 全核 CPU 压测 1 小时<br />
stress-ng --cpu $(nproc) --timeout 3600 --metrics<br />
<br />
# 指定算法<br />
stress-ng --cpu $(nproc) --cpu-method all --timeout 1800 --metrics<br />
<br />
# CPU + 内存联合压测<br />
stress-ng --cpu $(nproc) --vm 4 --vm-bytes 80% --timeout 3600</th>
</tr>
</thead>
<tbody>
</tbody>
</table>

| **配套监控** | **目的** | **常看指标** |
|:---|:--:|:--:|
| mpstat -P ALL 1 | 看是否均衡吃满 | 单核利用率、热点核心 |
| ipmitool sensor list | 看温度与风扇响应 | CPU Temp、Fan RPM |
| scaling_cur_freq / cpupower | 看是否降频 | 频率掉点 |

**5. 存储与 RAID 测试工具**

存储测试最容易“看起来做了很多，实际上结论不可靠”。原因通常有四个：没有预处理；没有说明块大小、队列深度、读写比例；没有避开文件系统缓存；没有用 iostat / SMART / 控制器状态交叉验证。

**5.1 fio：存储性能基线标准工具**

fio 是块存储与文件系统 I/O 测试的标准工具。使用 fio 时必须先讲清楚测试模型：顺序还是随机、读还是写、块大小、iodepth、numjobs、直通与否、是否 time_based、是否预处理。

</colgroup>
<thead>
<tr>

<br />
# 4K 随机读<br />

| **你当前环境的特别提醒** 你有两套 RAID 工具链：2U 华为服务器的 SP686 系列走 hiraidadm；6U 服务器的 Broadcom 卡走 storcli64。以后写测试结论时，必须把“控制器型号 + 工具链”一起写清，不要混用。 |
|----|

**5.5 存储测试结果如何写才专业**

- 不要只写“顺序读 3.5GB/s，正常”，要写块大小、队列深度、并发数、是否预处理、是否直通。

- 阵列盘必须同时给出逻辑盘状态与成员盘状态，不要只给逻辑盘一个 Optl 就结束。

- 性能异常必须同时给出 SMART/NVMe 错误、控制器事件、内核日志三类证据。

- 若是企业 SSD，应区分新盘峰值、写满后稳态、混合负载表现。

**7.4 固件与版本控制原则**

- 测试报告里必须写 BIOS、BMC、RAID FW、NIC FW、盘 FW、NPU/GPU FW、驱动版本。

- 升级前先导出版本与配置快照；升级后重新采集基线并验证业务影响。

- 批量平台不要只记录“最新版”，要记录“测试通过版本组合”。

- 固件升级涉及上下电、控制器重置或数据风险时，必须在 SOP 中单独列出前提与回退方案。

**8. 加速器与异构计算测试工具**

传统服务器整合测试以 CPU、存储、网络为主，但现在很多平台把 NPU/GPU、DPU、SmartNIC 纳入交付范围。测试思路的核心并没有变：先识别，再看拓扑，再看链路，再做温度/功耗/算力/稳定性验证。

**8.1 昇腾平台：npu-smi / ascend-dmi**

对你当前环境，npu-smi 是日常状态、温度、功耗、利用率的主入口；ascend-dmi 更偏向算力达成率验收。

- 先用 lspci 与 NUMA 归属确认卡是否在预期槽位和节点。

**8.2 行业通用补充：nvidia-smi / rocm-smi**

如果以后接触 GPU 平台，可以把 nvidia-smi 和 rocm-smi 理解为与 npu-smi 类似的设备健康入口。测试逻辑相同：识别、固件/驱动版本、温度、功耗、利用率、显存占用、错误日志。

|  亲和性问题  | 单卡正常，多卡异常 | numactl / hwloc / 中断与 PCIe 归属 |

**10. 结合你当前环境的落地建议**

这一章不是替代通用方法，而是把通用方法映射到你现在最常碰到的鲲鹏 + 昇腾 + 两套 RAID 工具链环境。

| **环境对象** | **你应优先用的工具** | **重点关注** |
|:---|:---|:---|
| 鲲鹏 920 双路 / 4 NUMA | lscpu, numactl, hwloc, ipmitool | 节点距离、频率、温度、功耗 |
| Atlas / 昇腾 NPU | lspci, npu-smi, ascend-dmi | NUMA 归属、链路速率、温度、达成率 |
| 2U 华为 RAID（SP686 系列） | hiraidadm, smartctl, dmesg | 控制器状态、盘状态、日志与固件 |
| 6U Broadcom RAID | storcli64, smartctl, dmesg | 阵列状态、成员盘、事件日志、重建进度 |
| 25GbE/聚合链路 | ethtool, iperf3, interrupts, bonding | 速率、错误包、流数、LACP 状态 |

- 鲲鹏平台要比普通 x86 更重视 SoC 集成与 NUMA/PCIe 共同作用，不要把网络、存储、NPU 当成完全独立模块去看。

- 你的 2U 与 6U 服务器不能共用同一 RAID 排查模板；报告中必须指明控制器厂家和工具名。

- NPU 与 NVMe 性能异常时，一定要补做 PCIe 链路速率与 NUMA 归属核查，这在你的环境里属于高频原因。

- 日志习惯继续保持：测试前清 dmesg/SEL，测试后留档。这一点你现有手册已经做得很好，建议固化成脚本。

**11. 测试前 / 中 / 后标准 SOP**

**11.1 测试前**

- 确认 BIOS / BMC / 控制器 / 盘 / NIC / 加速卡固件版本，并写入基线。

- 确认电源模式、风扇策略、CPU governor、IRQ 策略、MTU、LACP、RAID 状态。

- 清理 dmesg 与 SEL，保存空载温度/功耗/传感器快照。

- 确认测试目标盘、测试目标网口、目标设备 NUMA 归属，避免误测系统盘或业务口。

**11.2 测试中**

- 性能工具和监控工具同时开，不要只留最终跑分。

- 出现异常立即记时间点，并补采 dmesg、SEL、控制器日志、传感器快照。

- 长时间压测需定时采集功耗与温度，确认是否收敛而不是只看瞬时峰值。

**11.3 测试后**

- 导出 dmesg、journalctl、SEL、sensor、controller logs、性能原始输出。

- 将配置、版本、命令口径、结果、异常说明写入同一目录，保证可复盘。

- 对异常样本做“复现实验”和“对照实验”，不要只给一次性现象。

**12. 验收口径、报告字段与速查表**

**12.1 验收口径建议**

|  **维度**  | **建议验收口径**                                        |
|:----------:|:--------------------------------------------------------|
| 识别完整性 | 所有关键部件数量、型号、容量、链路与规划一致            |
| 功能可用性 | 链路 up、阵列正常、传感器正常、无关键 SEL/内核报错      |
|  性能基线  | 与规格书、历史稳定样本或同平台对照样本一致              |
|   稳定性   | 长压过程中无 critical error、无掉盘/掉卡/降级、温度收敛 |
|  功耗散热  | 峰值在供电和热设计范围内，风扇响应合理                  |
|  可追溯性  | 版本、命令、原始输出、异常日志齐全                      |

**12.2 报告最少字段**

- 平台信息：机型、CPU、内存、盘、网卡、加速卡、控制器、BMC。

- 软件信息：OS、内核、BIOS、BMC、驱动、固件版本。

- 测试口径：命令、参数、时长、负载模型、环境温度、电源策略。

- 结果摘要：通过/失败/风险项。

- 原始证据：日志目录、控制器日志、性能原始输出、截图或快照。

**12.3 场景速查表**

| **现象** | **先看什么** | **再看什么** | **常见归因** |
|:--:|:---|:---|:---|
| NVMe 跑分低 | lspci LnkSta | fio + iostat + smartctl | PCIe 降速、盘热降频、负载口径不对 |
| RAID 阵列降级 | 厂商 RAID CLI | smartctl / dmesg | 成员盘故障、重建中、foreign 配置 |
| 网络带宽低 | ethtool / iperf3 | interrupts / CPU 使用率 | 流数不足、协商异常、绑核问题 |
| 长压后掉速 | 温度/功耗曲线 | SEL / dmesg | 热边界、供电边界、固件策略 |
| NPU 达成率低 | npu-smi / ascend-dmi | lspci / NUMA / 温度 | 链路、亲和性、过热降频 |
| 系统偶发报错 | dmesg / journalctl | SEL / controller logs | 硬件抖动、驱动问题、边界触发 |
