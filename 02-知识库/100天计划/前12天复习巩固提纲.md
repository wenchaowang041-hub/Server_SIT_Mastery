# 前12天复习巩固提纲

## 适用场景

适合下班后没有现场机器、无法远程实操时，用来快速复习 Day1 到 Day12 的核心内容。

目标不是重新细看 12 篇全文，而是把前 12 天压缩成一条完整主线。

## 复习总目标

前 12 天真正建立的是 3 个能力：

1. 看懂服务器整机结构
2. 看懂关键数据路径
3. 建立存储和网络的基础性能直觉

## 第一轮：整机基础认知

对应：

- Day1：Linux 基础与硬件查看
- Day2：服务器架构核心（NUMA + PCIe + 网络）
- Day3：服务器性能分析，学会判断服务器瓶颈
- Day4：PCIe 架构与服务器设备拓扑

### 你要能回答的问题

1. 为什么一台服务器里会有 NUMA，而不是所有 CPU 和内存都完全等价？
2. PCIe 在服务器里解决的是什么问题？
3. 为什么性能分析不能只看 CPU 利用率？
4. 一块 NVMe、网卡、NPU 为什么都要先看 PCIe 拓扑？

### 这一轮你必须记住的结论

- NUMA 决定了“CPU 访问哪块内存更近”
- PCIe 决定了“设备挂在哪里、数据经过哪条路径”
- 性能瓶颈判断必须看 CPU、内存、IO、网络，而不是单点指标
- 不懂拓扑，后面的性能和兼容性分析很容易失真

## 第二轮：存储体系复习

对应：

- Day5：存储系统全景
- Day6：NVMe 架构深挖
- Day7：RAID 控制器与逻辑盘
- Day8：NVMe 顺序读写基线
- Day9：NVMe 4K 随机读写基线
- Day10：存储自动检测脚本

### 你要能回答的问题

1. NVMe、SATA、SAS、RAID 的关系分别是什么？
2. NVMe 为什么比传统 SATA/SAS 路径更快？
3. 什么是 controller，什么是 namespace？
4. RAID 卡、逻辑盘、物理盘、OS 块设备之间是什么映射关系？
5. 为什么顺序读写和 4K 随机读写看的是两种完全不同的能力？

### 这一轮你必须记住的结论

- 存储不是“盘”这么简单，而是协议、控制器、路径、逻辑抽象一起组成的系统
- NVMe 快，不只是盘快，而是协议轻、队列深、并发高、路径短
- RAID 卡会把底层物理盘重新抽象给 OS，看设备时不能只看 `lsblk`
- 顺序带宽和随机 IOPS 不能混着解释
- 自动化脚本的价值是把“人工识别”变成“可重复检查”

## 第三轮：网络基础复习

对应：

- Day11：网卡与链路基础
- Day12：网络吞吐基线

### 你要能回答的问题

1. 一张 4 口网卡为什么会在系统里看到多个接口？
2. `ip link`、`ethtool`、`ethtool -i` 分别看什么？
3. `iperf3`、`sar -n DEV 1`、`ip -s link` 三者分别观察哪一层？
4. 为什么链路是 `UP`，也不等于性能一定正常？

### 这一轮你必须记住的结论

- 网卡识别要同时看接口、驱动、速率、bus-info
- 网络吞吐测试不是只看 `iperf3` 数字，还要和接口统计对齐
- `sar -n DEV` 更偏系统视角，`ip -s link` 更偏接口计数视角
- 网络能通只是起点，不代表路径已经健康

## 第四轮：把前12天串成一台机器

这是最关键的一轮。

你要尝试用自己的话，把下面这条链完整说出来：

```text
CPU / NUMA
-> PCIe 拓扑
-> NVMe / RAID / NIC
-> OS 识别
-> 存储与网络基线
-> 初步性能判断
```

如果这条链你能说顺，说明前 12 天不是碎片知识，而是已经开始形成系统视角。
