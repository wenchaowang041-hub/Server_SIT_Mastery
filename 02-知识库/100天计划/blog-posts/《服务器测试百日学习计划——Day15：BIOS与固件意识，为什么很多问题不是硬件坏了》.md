**不是所有问题都叫硬件问题。**

## 一、为什么 Day15 很关键

你在现场一定会遇到这种情况：

- 同一台机器，换了个 BIOS 设置后，CPU 数量看起来变了
- 同一张卡，换个固件版本后，识别行为就不一样
- 同一台服务器，OS 一升级，驱动突然不稳定
- 同样的硬件组合，在一台机器上正常，在另一台机器上异常

如果你没有“分层意识”，这些问题最后都会被简单归成一句：

> 可能是硬件坏了。

这句话最大的问题不是“错”，而是**太粗**。

Day15 要你做的，就是把这句话拆开。

---

## 二、先把服务器的五层分清楚

你可以先把服务器理解成这 5 层：

```text
硬件
-> BIOS / BMC / Firmware / CPLD
-> Driver
-> OS
-> Tool / 应用
```

### 1. 硬件层

包括：

- CPU
- 内存
- NVMe
- 网卡
- NPU
- 主板
- PSU

### 2. 固件层

包括：

- BIOS
- BMC
- NVMe 固件
- 网卡固件
- RAID 卡固件
- CPLD 等板级版本

### 3. 驱动层

包括：

- 网卡驱动
- NVMe 驱动
- NPU 驱动
- 各类内核模块

### 4. OS 层

包括：

- Linux 内核
- systemd
- NetworkManager
- 块设备、网口、进程和调度

### 5. 工具层

包括：

- `lscpu`
- `ethtool`
- `ipmitool`
- `fio`
- `iperf3`
- `npu-smi`

> 以后排障时，第一反应不是“是不是坏了”，而是“这是哪一层的问题”。  

---

## 三、为什么分层意识这么重要

因为同一个现象，可能来自完全不同的层。

例如：

### 场景1：CPU 数量不对

可能是：

- BIOS 里 SMT / 线程配置改变了
- NUMA 配置变化了
- OS 暴露方式变了

### 场景2：网卡性能变差

可能是：

- BIOS 里的 PCIe 相关配置变化
- 固件版本不匹配
- 驱动不稳定
- IRQ / NUMA 路径不合理

### 场景3：NPU 工具认不到设备

可能是：

- PCIe 层没枚举
- 驱动没加载
- 固件不匹配
- 工具链版本不对

这就是为什么你不能一看到异常，就立刻说：

- 卡坏了
- 板子坏了
- 机器坏了

因为现象只是结果，问题可能发生在不同层。

## 四、BIOS 典型会影响什么

很多人把 BIOS 理解成“开机按 Delete 进去的设置界面”，但在服务器里，BIOS 其实会直接影响你在 OS 里看到的很多结果。

### 1. CPU 线程数

最典型的就是 SMT / 超线程开关。

如果 BIOS 里把 SMT 打开或关闭，`lscpu` 看到的逻辑 CPU 数就会变化。

例如本来：

- 每路 64 核
- SMT 打开
- 每核 2 线程

那逻辑 CPU 数就会翻倍。

### 2. NUMA 结构和暴露方式

BIOS 配置会影响：

- NUMA 是否启用
- NUMA 的表现形式
- 某些平台下的分区方式

所以你在系统里看到的 NUMA 结果，并不只是“硬件天生如此”，也可能是 BIOS 决定的。

### 3. PCIe 行为

这也是整机 SIT 里非常高频的一块。

BIOS 相关设置可能会影响：

- PCIe 设备枚举
- 链路速率
- bifurcation
- 插槽行为
- 某些卡的稳定性

也就是说，**PCIe 问题不一定是卡坏，也可能是 BIOS 设置或固件版本影响。**

### 4. 功耗与性能模式

不同 BIOS 功耗模式会直接影响：

- 频率表现
- 峰值性能
- 温度表现
- 长时间稳定性

所以很多“性能低”“温度高”“压测掉速”的问题，也不能只怪 OS。

## 五、现场里最常见的误区

### 误区1：设备不识别 = 硬件坏了

错。  
先问自己：

- BIOS / BMC / FW 版本对齐了吗？
- 驱动和 kernel 匹配吗？
- 工具链版本对吗？

### 误区2：性能差 = 网卡/盘/NPU 本身不行

错。  
也可能是：

- BIOS 功耗模式
- NUMA 路径
- IRQ 分布
- PCIe 行为

### 误区3：工具认不到 = 设备不存在

错。  
工具认不到，很多时候只是：

- 驱动没起来
- 固件不匹配
- 工具链和驱动版本不对

## 六、Day15 的核心方法：先问“它属于哪一层”

以后你看到一个问题，建议先过这 4 个问题：

1. 这会不会是工具层看到的表象？
2. 会不会是 OS / kernel 层兼容性问题？
3. 会不会是驱动层问题？
4. 会不会是 BIOS / BMC / FW / CPLD 这类固件层问题？

如果这些都排完了，再去判断是不是硬件本体故障。

> 这就是 Day15 最重要的能力：  
> **先分层，再下结论。**

---

## 七、你在现场最该记住的版本矩阵

以后不要单看某一个版本，要学会按“组合”看版本：

```text
BIOS
+ BMC
+ Device FW
+ CPLD
+ Driver
+ Kernel
+ OS
```

这几个东西只看单点意义不大，真正影响行为的是：

**它们组合在一起是否匹配。**

比如：

- 驱动版本对，但 kernel 太新
- BIOS 对，但 BMC 太旧
- 卡的 FW 对，但 CPLD 版本不对

这些都可能导致“设备存在，但行为异常”。
