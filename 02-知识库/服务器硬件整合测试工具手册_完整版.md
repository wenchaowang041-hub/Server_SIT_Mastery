> **第一章 硬件识别与基础信息查看**

上机第一步：摸清服务器硬件配置。本章覆盖从CPU架构、PCIe设备到存储接口的完整硬件识别方法。

**1.1 lscpu — CPU拓扑信息**

lscpu 读取 /proc/cpuinfo 和 sysfs，一条命令输出 CPU 架构、核数、NUMA 拓扑等全部关键信息。

> \# 查看完整CPU信息
>
> lscpu
>
> \# 只看关键字段
>
> lscpu \| grep -E 'Architecture\|CPU\\s\\\|Socket\|NUMA\|MHz\|Thread'
>
> \# 查看NUMA节点拓扑
>
> numactl --hardware
>
> \# 以JSON格式输出（便于脚本处理）
>
> lscpu -J

| **字段**           | **含义**    | **x86典型值**    | **ARM/鲲鹏典型值**  |
|--------------------|-------------|------------------|---------------------|
| Architecture       | 指令集架构  | x86_64           | aarch64             |
| CPU(s)             | 逻辑CPU总数 | 128（Xeon双路）  | 128（鲲鹏920双路）  |
| Thread(s) per core | 每核线程数  | 2（Intel超线程） | 1（鲲鹏无超线程）   |
| NUMA node(s)       | NUMA节点数  | 2（标准双路）    | 4（鲲鹏920双路）    |
| CPU MHz            | 当前频率    | 2100-3800 MHz    | 2600 MHz（鲲鹏920） |

**1.2 lspci — PCIe设备全景**

lspci 列出所有PCIe设备，是识别服务器整体配置最全面的命令。LnkSta速率对比是必查项。

> \# 过滤关键设备
>
> lspci \| egrep -i 'Non-Volatile\|RAID\|Network\|Accelerat\|SATA\|SAS\|Infini'
>
> \# 查看PCIe链路速率（检查降速）
>
> lspci -vv -s \<BDF\> \| grep -E 'LnkCap\|LnkSta'
>
> \# 树形拓扑
>
> lspci -tv
>
> \# 带设备ID（便于查驱动）
>
> lspci -nn

| **PCIe版本** | **单Lane速率** | **x4带宽** | **x8带宽** | **x16带宽** | **典型应用** |
|----|----|----|----|----|----|
| Gen3 | 8 GT/s | ~4 GB/s | ~8 GB/s | ~16 GB/s | SATA控制器、万兆网卡 |
| Gen4 | 16 GT/s | ~8 GB/s | ~16 GB/s | ~32 GB/s | NVMe SSD、昇腾NPU、25G网卡 |
| Gen5 | 32 GT/s | ~16 GB/s | ~32 GB/s | ~64 GB/s | 最新NVMe、AI加速卡 |

> *⚠️ LnkCap=设备支持上限，LnkSta=实际协商速率。Cap Gen4但Sta Gen3说明PCIe降速，直接导致NVMe/NPU性能减半，必须在BIOS中将PCIe Slot设为Gen4强制模式。*

> **第二章 CPU 测试工具**

**2.1 stress-ng — CPU综合压力测试**

stress-ng 是最常用的CPU压测工具，支持200+种压测方法，是服务器SIT测试的标准选择。

> \# 安装
>
> yum install stress-ng -y
>
> \# 全核满载（最标准的压测方式）
>
> stress-ng --cpu \$(nproc) --timeout 3600
>
> \# 指定压测方法（fma=浮点乘加，最接近真实AI计算）
>
> stress-ng --cpu \$(nproc) --cpu-method fma --timeout 3600
>
> \# 同时压CPU+内存（全系统压测）
>
> stress-ng --cpu \$(nproc) --vm 4 --vm-bytes 80% --timeout 7200
>
> \# 只压特定NUMA节点
>
> numactl --cpunodebind=0 stress-ng --cpu 32 --timeout 3600
>
> \# 带详细性能指标输出
>
> stress-ng --cpu \$(nproc) --metrics --timeout 3600 --log-file stress.log

**2.2 HPL / LINPACK — 浮点运算能力标准基准 ★**

HPL（High Performance Linpack）是 TOP500 超级计算机排名使用的基准测试，通过求解大规模稠密线性方程组测量系统浮点峰值性能（GFLOPS/TFLOPS）。在服务器交付验收中常作为**计算性能达成率**的权威依据。

**▸ 工作原理**

HPL 对 N×N 双精度浮点矩阵进行 LU 分解，计算量约为 2/3·N³ 次浮点运算。通过实际运算时间计算 Gflops，与理论峰值对比得出达成率。

**▸ 安装**

> \# 依赖：OpenMPI + BLAS库
>
> yum install openmpi openmpi-devel blas blas-devel -y
>
> \# 下载HPL源码
>
> wget http://www.netlib.org/benchmark/hpl/hpl-2.3.tar.gz
>
> tar xf hpl-2.3.tar.gz && cd hpl-2.3
>
> \# 配置编译（ARM/鲲鹏用OpenBLAS，Intel用MKL）
>
> \# 安装OpenBLAS
>
> yum install openblas-devel -y
>
> \# 编译
>
> cp setup/Make.Linux_PII_CBLAS_gm .
>
> make arch=Linux_PII_CBLAS_gm
>
> \# 二进制文件生成在 bin/Linux_PII_CBLAS_gm/xhpl

**▸ HPL.dat 关键参数配置**

> \# HPL.dat 示例（512GB内存双路服务器）
>
> HPLinpack benchmark input file
>
> 1 \# of problems sizes (N)
>
> 300000 Ns \# N = sqrt(0.88 \* 内存字节数 / 8)
>
> 1 \# of NBs
>
> 192 NBs \# 块大小：x86用192/384，ARM用128/256
>
> 0 PMAP process mapping (0=Row-, 1=Column-major)
>
> 1 \# of process grids (P x Q)
>
> 2 Ps \# P: MPI进程行数
>
> 2 Qs \# Q: MPI进程列数（P×Q = MPI进程总数）
>
> 16.0 threshold
>
> 1 \# of panel fact
>
> 2 PFACTs (0=left, 1=Crout, 2=Right)
>
> 1 \# of recursive stopping criterium
>
> 4 NBMINs
>
> 1 \# of panels in recursion
>
> 2 NDIVs
>
> 📋 N的计算公式：N = sqrt(内存总字节 × 88% / 8)。例如512GB内存：N = sqrt(512×1024³×0.88/8) ≈ 244,000。P×Q应等于MPI进程数，且尽量接近正方形（如2×2、4×4）。

**▸ 运行命令**

> \# 启动HPL（4进程）
>
> mpirun -np 4 ./xhpl
>
> \# 绑核运行（推荐，避免进程迁移）
>
> mpirun -np 4 --bind-to core ./xhpl
>
> \# 典型输出
>
> \# T/V N NB P Q Time Gflops
>
> \# WR11 300000 192 2 2 3542.1 5.682e+03
>
> \# 5682 Gflops = 5.682 Tflops（双精度浮点）

| **处理器** | **核数** | **理论峰值（FP64）** | **HPL典型达成率** |
|----|----|----|----|
| Intel Xeon Gold 6338（单路） | 32核 | 2.867 Tflops | ~85-92% |
| AMD EPYC 7763（单路） | 64核 | 5.170 Tflops | ~88-93% |
| 鲲鹏920（单路） | 64核 | 理论值依频率 | ~80-88%（ARM优化后） |
| 双路服务器 | ×2 | 上述×2 | 相同达成率 |

> *⚠️ HPL 会占用接近100%的CPU和内存，运行期间服务器几乎无法响应其他请求。单次测试耗时通常30分钟~2小时。运行前必须清空SEL日志并确认散热正常。*

**2.3 perf — Linux性能分析框架 ★**

perf 是 Linux 内核自带的性能分析工具，通过硬件性能计数器（PMU）采集 CPU 事件，可以精确分析 CPU 的 IPC（每时钟指令数）、Cache命中率、分支预测失败率等指标，是深度分析 CPU 性能瓶颈的必备工具。

**▸ 安装**

> \# 安装（与内核版本匹配）
>
> yum install perf -y
>
> \# 或从内核源码编译
>
> cd /usr/src/kernels/\$(uname -r)/tools/perf && make install

**▸ perf stat — 统计CPU性能指标**

> \# 统计某个命令的CPU性能指标
>
> perf stat ls /
>
> \# 对特定进程统计（30秒）
>
> perf stat -p \<PID\> sleep 30
>
> \# 指定统计事件
>
> perf stat -e cycles,instructions,cache-misses,branch-misses ./workload
>
> \# 全核统计（压测期间分析）
>
> perf stat -a sleep 30
>
> \# 典型输出解读：
>
> \# 1,234,567,890 cycles \# 总时钟周期
>
> \# 2,345,678,901 instructions \# 总指令数
>
> \# 1.90 insn per cycle \# IPC（越高越好，\>2为优秀）
>
> \# 12,345,678 cache-misses \# Cache未命中次数

**▸ perf top — 实时热点函数**

> \# 实时查看CPU热点函数
>
> perf top
>
> \# 只看内核态热点
>
> perf top -K
>
> \# 只看特定进程
>
> perf top -p \<PID\>
>
> \# 指定采样频率
>
> perf top -F 999

| **perf指标**         | **含义**             | **参考值**               |
|----------------------|----------------------|--------------------------|
| IPC (insn per cycle) | 每时钟周期执行指令数 | \>2.0优秀，\<1.0存在瓶颈 |
| cache-miss rate      | LLC缓存未命中率      | \<5%正常，\>20%内存瓶颈  |
| branch-miss rate     | 分支预测失败率       | \<1%正常，\>5%代码问题   |
| cycles               | 总时钟周期           | 结合时间计算实际频率     |

**2.4 turbostat — CPU频率和功耗实时监控 ★**

turbostat 由 Intel 工程师开发，读取 MSR 寄存器，报告每个 CPU 核心的实际频率、C-state停留率、温度和功耗。是验证 Turbo Boost 是否正常工作的最权威工具。鲲鹏/ARM平台可用 cpupower 替代。

> \# 安装（内含在 kernel-tools 包）
>
> yum install kernel-tools -y
>
> \# 每1秒报告所有核心的频率和功耗
>
> turbostat --interval 1
>
> \# 只显示关键指标（简洁模式）
>
> turbostat --quiet --interval 1 --show Busy%,Avg_MHz,Bzy_MHz,CoreTmp,PkgWatt
>
> \# 对特定命令进行分析
>
> turbostat stress-ng --cpu \$(nproc) --timeout 60
>
> \# 输出字段说明：
>
> \# Avg_MHz - 平均频率（包含空闲时间）
>
> \# Bzy_MHz - 忙时实际频率（Turbo频率）
>
> \# Busy% - CPU忙碌时间百分比
>
> \# CoreTmp - 每核温度（°C）
>
> \# PkgWatt - 整包功耗（瓦特）
>
> 📋 ARM/鲲鹏平台不支持 turbostat（基于 Intel MSR），请改用：watch -n 1 'cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_cur_freq' 查看频率，ipmitool sensor 查看温度。

**3.2 memtest86+ — 裸机内存检测 ★**

memtest86+ 是独立运行于裸机（不依赖OS）的内存检测工具，在启动阶段运行，能检测出OS下工具无法发现的内存硬件缺陷，是内存交付验收的黄金标准。

> \# 安装到GRUB引导菜单
>
> yum install memtest86+ -y
>
> grub2-mkconfig -o /boot/grub2/grub.cfg
>
> \# 开机时在GRUB菜单选择 'Memory Test (memtest86+)'
>
> \# 或使用USB启动盘方式
>
> \# 从官网下载最新版本（更推荐）
>
> \# https://www.memtest86.com/download.htm
>
> \# 版本7.x支持DDR5和ECC内存
>
> \# 推荐运行时间：
>
> \# 快速检查：至少完成1个完整pass（约30分钟-2小时）
>
> \# 严格交付验收：至少2个完整pass（约1-4小时）

| **测试Pass** | **完成度** | **检测内容**                         |
|--------------|------------|--------------------------------------|
| Pass 1       | 基础检测   | 地址线、数据总线故障，大多数简单故障 |
| Pass 2       | 进阶检测   | 细微位翻转错误，温度相关故障         |
| Pass 3+      | 完整验证   | ECC性能、随机访问故障，交付验收标准  |

> *⚠️ 发现错误时记录：错误地址（Address）、预期值（Expected）和实际值（Actual）。连续出错地址段对应物理内存槽位，需更换该槽位的内存条。*

**3.3 STREAM — 内存带宽标准基准**

>
> \# 关注 Triad 综合带宽（MB/s）
>
> \# 鲲鹏920双路参考值：~190,000+ MB/s

| **错误类型** | **含义** | **处理** | **紧急程度** |
|----|----|----|----|
| CE（Correctable Error） | 单bit翻转，ECC自动纠正 | 记录频率，高频率需更换 | 中 |
| UE（Uncorrectable Error） | 多bit翻转，无法纠正 | 立即停机更换内存条 | 高 |
| Scrub Error | 内存巡检发现的错误 | 同CE处理 | 中 |
| Address Parity | 地址校验失败 | 更换内存，可能槽位问题 | 高 |

> \# Step 1: 确认当前频率
>
> cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_cur_freq
>
> \# 或：turbostat --show Bzy_MHz --interval 1
>
> \# Step 2: 查温度（热降频最常见）
>
> ipmitool sensor list \| grep -i cpu
>
> \# Step 3: 查调频策略
>
> cpupower frequency-info -p
>
> \# Step 4: 设为性能模式排除
>
> cpupower frequency-set -g performance

**▸ NPU达成率低**

> \# Step 1: 查NPU温度（热降频）
>
> npu-smi info -t common \| grep Temp
>
> \# Step 2: 查PCIe链路（速率降速）
>
> lspci -vv -s \<NPU BDF\> \| grep LnkSta
>
> \# Step 3: 查驱动版本
>
> npu-smi info \| grep Driver
>
> \# Step 4: 查NPU错误
>
> npu-smi info -t error
