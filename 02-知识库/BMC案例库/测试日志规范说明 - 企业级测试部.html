测试日志规范说明

1. 总则

1.1 目的

为规范服务器测试全流程的日志记录与归档管理，确保测试数据的真实性、完整性、可追溯性，降低问题定位难度，提升测试协作效率，同时满足内部审计、客户交付及合规性要求，特制定本规范。本规范适用于服务器产品的所有测试场景，包括功能测试、性能测试、稳定性测试、兼容性测试等，全体测试工程师需严格遵照执行。

1.2 适用范围

本规范覆盖测试过程中的所有日志相关活动，包括日志的采集、记录、整理、命名、存储、归档、检索及留存等环节；适用于所有服务器测试项目及相关测试人员。

1.3 核心原则

  • 可追溯性：每一条日志均需关联测试用例、测试时间、测试人员、测试环境等关键上下文信息，确保问题可回溯、过程可复现。

  • 完整性：全面记录测试过程中的操作、数据、现象及结果，不遗漏关键信息，满足问题定位和复盘分析需求。

  • 规范性：统一日志记录格式、命名规则及归档方式，确保日志管理的标准化、规范化，提升协作效率。

  • 高效性：优化日志采集与归档流程，平衡日志完整性与存储/检索效率，避免无效日志冗余。

2. 日志存档

2.1 内部测试

测试报告不强制要求归档，但在测试管理系统-报告页面，必须完成测试报告整理工作。

1）测试管理系统中提交关键测试过程信息说明和相应截图，通过执行历史可以识别测试行为和结果的正确性。

2.2 外部测试

测试报告必须归档，以提交给客户确认的最终版本为准。

  • 优先参考外部客户测试日志规范，进行测试日志归档整理

3 日志规范

3.1 截图（适用于执行类）

  • 截图插入在系统或报告等其他载体中时，要求截图清晰可辨（禁止长截图或多图合成，会出现上传图片不清晰问题），体现完整的测试关键信息

  • 截图少于3屏时，保留所有测试过程截图

  • 截图超过3屏时，保留至少3张核心截图，分别为：

    • 包含测试执行命令的截图（明确操作对象及指令）；

    • 中间关键信息部分截图（如动态性能数据、测试正常运行标志、关键交互信息等）；

    • 最后信息结束部分截图（如信息汇总、执行结果提示页）；

    • 选择性通过绘图工具添加标注说明，文字清晰可辨，截图信息明确指向

  • 截图以文件形式保存时，参考本文测试日志命名规范进行保存

3.2 截图（适用于信息类）

注：信息类截图通常针对系统配置、环境状态、硬件拓扑或静态日志等非动态执行过程，侧重于信息的准确传达与可追溯性。

  • 画质与呈现底线

    • 截图嵌入系统或测试报告时，必须保证清晰可辨。严禁使用长截图或多图合成拼接（以规避平台压缩导致的画质受损），确保各项关键参数肉眼易读。

  • 上下文的双向关联

    • 拒绝“孤立截图”： 每张截图必须具备明确的业务上下文，严格与具体的测试步骤或预期结果建立一一映射关系。

    • 截图的上下文中需配有简短的引导语（例如：“步骤3：BIOS配置界面如下”或“预期结果：系统正确识别硬件拓扑”）。

  • 核心观察点的显性标注

    • 精准聚焦： 当界面包含海量信息时，必须使用标准化绘图工具（如醒目的红框、箭头）对具体观察点（如特定字段、关键参数、异常告警等）进行精准标识。

    • 辅助说明： 标识处需视情况添加清晰、简练的文字说明，直接点明该界面的核心结论（例如：“内存识别容量正常”、“SNC模式已开启”等），最大限度降低评审人员的信息提取成本。

  • 信息截取策略

    • 单屏涵盖全貌： 若能在单屏内展示，优先截取包含完整配置参数或系统状态的全局画面。

    • 多屏提炼核心： 若信息列表较长（超过3屏），无需逐页截图，仅保留能证明关键状态的代表性截图（如具有代表性的数据页、关键差异页或首尾页即可）。

  • 文件归档命名

    • 截图若以独立文件形式保存，必须严格参照当前的《测试日志命名规范》执行，确保文件名内嵌测试用例编号、关键步骤节点或时间戳，保障后期审计的快速溯源。

3.3 日志-系统原生

  • 保留原生日志状态，不需要更改名称（如messages、syslog等）

  • 保留原生测试工具生成的日志文件（如SPECCPU测试自动化成的result里的文件保持不变）

  • 如果测试跨周，注意将系统上一周的备份记录一同抓取归档，如下面的messages和messages-20260120包含了上周末的系统日志信息：

[root@bogon ~]# ls /var/log/
anaconda  btmp    ctyunos-security.log  dnf.rpm.log  hawkey.log           maillog            private  rpmpkgs-20260118  spooler              storelibdebugit.txt.1  up2date
audit     chrony  dnf.librepo.log       dracut.log   hawkey.log-20260118  messages           README   samba             sssd                 systemtap.log          wtmp
boot.log  cron    dnf.log               firewalld    lastlog              messages-20260120  rpmpkgs  secure            storelibdebugit.txt  tuned
[root@bogon ~]#

3.4 日志-自定义生成

  • 以执行命令或关键参数命名，做到见名知意效果，如：

    • ipmitool sel elist输出命名为：sel

    • lspci -vvvs 2a:00.0输出命名为：lspci_2a_00_0.txt

    • fio 队列深度32下的4k randread输出命名为：randread_32_sdb_4k.txt

  • 以用例名称或预期结果命名，如：

    • GPU在OS侧信息验证

    • GPU在BMC Web侧信息验证-BMC_03.07.03、GPU在BMC Web侧信息验证-BMC_03.08.03（针对不同版本差异化，需要增加版本标识）

    • OS_Cycles_20251201、OS_Cycles_20251205（针对不同时间复现数据，需要增加时间标识）

3.5 文件夹组织结构

  • 采用“测试编号/序号-测试用例”二级文件夹结构（测试报告以文件或压缩包形式保存归档），确保日志归档清晰有序，具体结构举例：

4 执行与监督

4.1 培训与宣贯

测试负责人组织组内测试人员进行宣贯培训，确保每个人理解规范并按要求执行。

4.2 日常监督

测试负责人需定期对测试日志的记录与归档情况进行检查，重点核查日志完整性、命名规范性、归档及时性及敏感信息脱敏情况；对违反规范的情况需及时督促整改，并记录整改结果。

4.3 持续优化

本规范将根据测试工作实际需求、技术迭代及最佳实践更新情况，会不定期进行评审与优化，确保规范的适用性和有效性。

5 附则

本规范由测试部门负责解释，自发布之日起正式执行。原有测试日志相关规定与本规范不一致的，以本规范为准。
