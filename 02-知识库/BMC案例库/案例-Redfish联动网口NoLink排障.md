# Day13：Redfish 联动网口 NoLink 排障

## 今日主题

把一台“BMC 可访问、但 OS 业务网口全部不通”的机器，从管理面可见推进到业务口恢复可用。

这一课的重点不是配 IP，而是训练你在整机 SIT 现场里，遇到网口全断链时的正确判断顺序。

## 先记住结论

当 BMC 正常、Redfish 能看到网卡和端口、但 OS 侧网口全部 `NoLink` / `Disconnected` 时，优先排查物理层、槽位适配、网卡类型和拓扑，不要先陷入 OS 配置。

## 一、这类问题为什么容易误判

很多人看到网不通，第一反应就是：

- 没配 IP
- NetworkManager 有问题
- 驱动没起来
- 系统配置错了

但如果 Redfish 已经能枚举出网卡和端口，说明：

- 管理面没有彻底失明
- 设备不一定没识别
- 问题可能不在 IP 层

也就是说，这时最该先确认的是：

- 网卡类型对不对
- 插槽位置对不对
- 线缆和光模块对不对
- 链路有没有真正起来

## 二、这次现场的典型现象

### 已知条件

- BMC IP 可访问
- Redfish Basic Auth 可用
- 可以访问 PCIe 和网卡相关资源
- OS 环境为 openEuler 22.03 (LTS-SP4)

### 现场表现

- Redfish 能看到两张网卡
- 一张是 `2*25GE` 光口卡
- 一张是 `4*GE` 电口卡
- 总共 6 个主机口
- 6 个主机口初始全部显示 `NoLink` / `Disconnected`

这类现象很关键，因为它说明：

- 不是单个口偶发异常
- 不是单纯某个 IP 没配
- 更像是整张卡、整类槽位或接线关系有问题

## 三、正确排障顺序

### 第1步：先确认管理面是不是正常

你先要确认 BMC 和 Redfish 是否工作正常。

核心目标：

- BMC 是否可访问
- Redfish 是否返回正常
- 能否读取网卡与 PCIe 信息

如果管理面都不通，那就不是今天这类问题，要先回到更底层的点亮和 BMC 检查。

### 第2步：确认网卡有没有被管理面识别

你要确认的不是 OS 有没有 `eth0`，而是整机有没有识别到板卡。

重点看：

- PCIe 设备有没有枚举
- 网卡资源有没有出现在 Redfish 里
- 是几张卡
- 每张卡分别是什么类型

这一层的结论决定后面怎么排：

- 如果 BMC/Redfish 都看不到卡，优先查板卡、供电、版本、插槽
- 如果 BMC/Redfish 能看到卡，但端口全 `NoLink`，优先查物理连接和槽位适配

### 第3步：区分网卡类型，不要把“有网卡”当成“可通网”

现场很容易犯的错是：

- 只看到有两张卡，就默认装对了

但实际要继续拆开看：

- 哪张是光口卡
- 哪张是电口卡
- 哪张应该插在哪个槽位
- 当前接的是电口还是光口

如果槽位存在适配约束，光卡和电卡插反后，表现出来的故障不一定是“卡消失”，而可能是：

- 卡还在
- 端口也在
- 但全部断链

这个现象非常迷惑新手。

### 第4步：看到全部 `NoLink` 时，先查物理层

优先排查顺序建议固定成：

1. 网卡类型是否正确
2. 槽位是否插对
3. 线缆是否插对
4. 光模块或电口连接对象是否正确
5. 对端交换机口是否正常
6. OS 驱动和配置是否正常

这一步的核心原则是：

全口 `NoLink` 时，优先查物理层和拓扑；单口异常时，再更关注端口本身和配置。

### 第5步：现场动作要和拓扑图联动

这类问题不能脱离接线拓扑来判断。

你要一边看：

- 当前槽位
- 当前卡类型
- 当前接线

一边对照：

- 项目要求的槽位关系
- 电口卡 / 光口卡的安装位置
- 业务口实际应接到哪里

如果拓扑图要求电卡和光卡分别占不同槽位，那就不能凭“都能插上”来判断装配正确。

### 第6步：问题闭环后，再回到 OS 配置

只有当链路真实起来了，才值得继续做：

- OS 静态 IP
- 路由
- 网关连通性
- 网络专项测试

否则你越往上层排，越容易浪费时间。

## 四、这次问题的真实根因

这次问题最终闭环结果是：

- 光卡和电卡槽位对调
- 槽位与网卡类型存在适配约束
- 调整到正确槽位后，业务口 `eno1` 恢复为 `UP`

这个结论的价值不只是“今天修好了”，而是你要记住一种典型模式：

管理面识别正常，不等于业务链路正常。  
设备存在，不等于安装关系正确。

## 五、这类问题的标准分析模板

以后你写复盘或面试表达时，可以直接按这个结构说：

### 1. 现象

- BMC 可访问
- Redfish 可读
- 网卡资源存在
- 所有主机口 `NoLink`

### 2. 初步判断

- 不是单纯 IP 配置问题
- 不是纯 OS 层问题
- 优先怀疑物理层、槽位适配、网卡类型和拓扑

### 3. 排查动作

- 查 Redfish 网卡与 PCIe 资源
- 区分光口卡和电口卡
- 对照接线拓扑
- 核对插槽位置
- 调整板卡安装位置

### 4. 闭环结果

- 光卡、电卡插反
- 调整后 `eno1` 恢复链路

### 5. 沉淀价值

- 后续项目装机时，必须把“槽位与网卡类型匹配关系”纳入 bring-up 检查项

## 六、你今天应该背下来的方法

遇到“BMC 通、OS 网口不通”时，先问自己这 5 个问题：

1. 管理面能不能看到网卡
2. 看到的是哪几类网卡
3. 这些卡插在什么槽位
4. 这些槽位是否允许这类卡工作
5. 链路没起来之前，我是不是过早去配了 OS IP

只要这 5 个问题没过清楚，就不要急着把精力放在网络配置上。

## 七、现场常用检查点

### 管理面

- BMC IP 是否可达
- Redfish 是否可访问
- PCIe 设备是否存在
- NetworkAdapter / EthernetInterface 信息是否完整

### 物理层

- 光卡 / 电卡类型
- 插槽位置
- 线缆连接位置
- 对端交换机口状态
- 光模块或电口连接介质

### OS 层

- `ip a`
- `ethtool <port>`
- `ethtool -i <port>`
- `nmcli device status`

但记住，OS 层不该排在最前面。

## 八、面试式自测题

### 题1

如果 Redfish 能看到网卡，但 6 个端口全部 `NoLink`，你最先排什么？

标准答案方向：

- 先排物理层、槽位适配、网卡类型和拓扑
- 不先陷入 OS IP 配置

### 题2

为什么“Redfish 能看到网卡”不能直接说明网络没问题？

标准答案方向：

- 只能说明管理面识别到了设备
- 不能说明业务链路已正常建立
- 也不能说明槽位、线缆和端口类型全部正确

### 题3

什么情况下你才应该开始配 OS 静态 IP？

标准答案方向：

- 至少要先确认某个业务口真实 `UP`
- 物理链路正常
- 网卡类型和槽位匹配正确

## 九、今天的补习结论

- 网口全断链时，最怕的是排障顺序错。
- 整机 SIT 现场里，配置动作要服从判断顺序，不能抢跑。
- Redfish 是很好的观察入口，但它解决不了物理层错误，必须和现场装配、拓扑、槽位关系一起看。
