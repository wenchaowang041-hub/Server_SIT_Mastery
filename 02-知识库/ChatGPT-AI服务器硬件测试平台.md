# AI服务器硬件整合测试 SIT 自动化平台（PRD + 可落地技术需求说明）

## 1. 文档定位

本文档用于指导 Codex / 开发团队实现一套**基于 AI 的服务器硬件整合测试 SIT 自动化平台**。该平台面向服务器整机硬件集成测试场景，核心目标是：

1. 将服务器从 **BOM / 配置接收 → 上架装机 → BMC/OS 接入 → 配置核验 → 用例推荐 → 测试执行 → 日志分析 → 缺陷提报 → 报告生成** 全流程标准化、平台化、自动化。
2. 覆盖 **鲲鹏平台、昇腾 NPU、华为 RAID 卡** 等重点对象，同时通过插件化机制兼容主流通用服务器硬件。
3. 让测试过程具备**证据链、可追溯、可审计、可复现、可沉淀**的能力。
4. 用 AI 提升用例推荐、日志分析、异常归因、报告总结和缺陷提报效率，但**最终结果要可验证、可追责，不能黑盒化**。

---

## 2. 背景与问题定义

当前服务器硬件 SIT 测试常见痛点：

1. **信息割裂**：BOM、拓扑接线图、BMC 配置、OS 配置、设备枚举信息、固件版本、测试记录分散在 Excel、文档、命令行和人工记录中。
2. **人工核对成本高**：装机后需人工核对 CPU、内存、磁盘、网卡、RAID、NPU、固件等是否与 BOM 一致，容易漏项。
3. **测试执行不标准**：不同人员执行步骤、命令、判定标准不一致，证据不统一，导致结果不可复核。
4. **日志收集无体系**：测试前后日志未标准收集，问题复盘依赖个人经验。
5. **缺陷提报效率低**：复现信息、版本信息、日志、截图、环境信息往往缺失，导致定位效率低。
6. **兼容性扩展差**：不同平台、不同品牌设备工具差异大，自动化脚本难复用。
7. **知识沉淀不足**：每次项目重复搭环境、重复分析，历史问题和处置经验难复用。

因此需要建设统一的平台，完成“**资产基线化、测试模板化、执行流程化、证据标准化、分析智能化、兼容插件化**”。

---

## 3. 产品目标

### 3.1 业务目标

1. 将整机 SIT 的核心流程线上化，减少人工 Excel/文档管理。
2. 实现服务器装机后**自动发现、自动核对、自动推荐测试、自动留痕、自动报告**。
3. 建立整机配置“冻结基线”，作为后续回归、问题定位、版本对比的基准。
4. 让问题从“现象”快速收敛到“配置差异 / 版本差异 / 硬件异常 / 日志证据”。

### 3.2 效率目标

1. BOM 与实机配置核对自动化率 ≥ 90%。
2. 测试前环境准备自动化率 ≥ 80%。
3. 测试执行证据完整率 ≥ 95%。
4. 缺陷提报平均耗时降低 ≥ 50%。
5. 报告生成时间从小时级降低到分钟级。

### 3.3 质量目标

1. 测试步骤规范统一，减少人为漏测、误判。
2. 每条测试用例、每一步测试动作均可追溯。
3. 每次问题分析可追溯到：机器、BMC IP、OS IP、BOM 版本、固件版本、测试版本、日志、证据、结论。

## 4. 适用范围

### 4.1 首期重点支持

1. **CPU 平台**：鲲鹏平台优先，兼容 x86 主流平台。
2. **加速卡**：昇腾 NPU 优先，兼容 GPU / 其他 PCIe 加速卡扩展。
3. **RAID 卡**：华为 RAID 卡优先，兼容 Broadcom / LSI 等主流 RAID 卡。
4. **磁盘**：NVMe、SATA、SAS。
5. **网络**：板载网卡、PCIe 网卡。
6. **管理面**：BMC。
7. **固件对象**：BIOS、BMC、RAID、NIC、SSD/NVMe、NPU 等。

### 4.2 场景范围

1. 新项目整机 bring-up 后硬件配置核验。
2. 量产前整机 SIT。
3. 固件升级前后回归测试。
4. 变更件替换后的兼容性 / 回归测试。
5. 异常机器复测与问题复现。

### 4.3 不在首期范围但需预留

1. 大规模机房调度系统替代。
2. 生产制造 MES 全流程替代。
3. 自动上架机器人 / 自动布线。
4. 全量 OTA 升级平台。

## 5. 用户角色与权限

### 5.1 角色定义

1. **测试工程师**
   - 录入 / 导入机器信息
   - 维护 BMC/OS 凭据
   - 启动测试任务
   - 查看结果、补充备注、提交缺陷

2. **测试主管 / 项目负责人**
   - 配置项目模板
   - 审批基线冻结
   - 查看项目整体进展、质量统计、风险汇总

3. **平台管理员**
   - 管理用户、角色、权限
   - 管理设备驱动/工具源/插件
   - 管理用例模板库、报告模板、缺陷模板

4. **开发 / 问题分析工程师**
   - 查看日志、证据、复现步骤、环境信息
   - 使用平台导出的 bug 包定位问题

5. **审计 / 质量管理人员**
   - 只读查看测试记录、证据、报告、变更历史

### 5.2 权限要求

1. 不同项目、不同机器数据需按项目隔离。
2. BMC/OS 密码必须加密保存，前端不可明文回显。
3. 高危操作（重启、升级、清盘、固件刷新）需细粒度权限控制。
4. 所有关键动作必须有审计日志：谁、在什么时间、对哪台机器、执行了什么。

## 6. 总体业务流程

### 6.1 端到端流程

1. 项目创建。
2. 导入项目 BOM、物料清单、配置信息、拓扑接线图。
3. 现场按照接线图完成装机。
4. 上电后配置 BMC IP，安装 OS，配置 OS IP。
5. 录入平台：BMC IP、OS IP、用户名、密码、项目、机型、序列号等。
6. 平台连接服务器：
   - 通过 BMC 获取带外信息。
   - 通过 SSH / Agent 获取 OS 带内信息。
7. 平台自动抓取基础配置：
   - CPU、内存、磁盘、RAID、网卡、PCIe、NPU、驱动、固件、系统版本等。
8. 平台执行多维对比：
   - BMC vs OS
   - 实机 vs BOM
   - 当前 vs 历史基线
9. 对比结果：
   - 一致：生成并冻结配置基线。
   - 不一致：标红、告警、生成待人工核对项。
10. 平台自动检测测试依赖工具，缺失则从受控源自动安装。
11. 工程师上传测试用例，或从平台用例库选择；平台结合配置信息推荐适合用例。
12. 平台按规范模板执行用例，自动记录步骤、日志、截图、命令输出、判定结果。
13. 每次测试前后自动收集系统日志与关键状态，做差异比对。
14. AI + 规则引擎分析异常日志，输出问题摘要、可能根因、关联版本、建议动作。
15. 如失败，平台生成 bug 提报包与 bug 模板。
16. 全部测试完成后，自动生成项目级 / 单机级测试报告。

## 7. 核心设计原则

1. **真实可信优先于炫技 AI**：所有 AI 结论必须可回溯到原始日志、命令输出或规则。
2. **模板先行**：所有测试用例、报告、bug 提报、证据采集都有固定模板。
3. **插件化适配**：不同品牌工具差异通过适配器处理，平台主流程不因品牌耦合。
4. **数据基线化**：每次关键测试前后形成快照。
5. **人机协同**：自动化做高频、标准动作；高风险结论保留人工复核入口。
6. **证据链完整**：结果、截图、日志、命令输出必须绑定到用例步骤。
7. **可扩展**：支持新硬件、新协议、新品牌、新用例类型快速接入。

# 9.3 配置采集与对比引擎

## 9.3.1 采集原则

每次采集都生成一份**结构化快照**，作为后续比对与追溯依据。

## 9.3.2 对比维度

1. **BMC vs OS 对比**
   - CPU 数量
   - 内存总量
   - 磁盘数量
   - 网卡数量
   - FRU/SN/PN
   - 固件版本
2. **实机 vs BOM 对比**
   - 型号、数量、容量、速率、接口、版本
3. **当前 vs 历史基线**
   - 固件变更
   - 部件替换
   - 驱动变化
   - 设备消失 / 新增
4. **测试前 vs 测试后**
   - 错误计数
   - 设备健康状态
   - dmesg / mcelog / npu-smi / 存储错误 / RAID 事件 / NIC error

## 9.3.3 输出结果

1. 一致项：绿色。
2. 可接受偏差项：黄色，标明原因。
3. 不一致项：红色，生成待核对任务。
4. 无法获取项：灰色，标记采集失败原因。

## 9.3.4 冻结机制

1. 首次核对通过后可人工审批“冻结为基线”。
2. 冻结项包括：硬件配置、关键版本、系统版本、驱动版本、拓扑摘要。
3. 后续所有测试结果均引用该基线。
4. 若发生部件替换 / 版本升级，需触发“基线变更流程”。

## 9.3.5 必须支持的对比策略

1. 精确匹配
2. 范围匹配（版本范围、容量范围）
3. 别名匹配（同型号不同字符串表示）
4. 替代件匹配
5. 品牌白名单匹配

# 9.4 工具依赖管理模块

## 9.4.1 功能目标

## 9.4.2 工具类型

4. 性能工具：stress-ng, fio, sysbench, perf, numactl

6. 厂商工具：BMC CLI、RAID CLI、固件查询工具等

## 9.4.3 行业补充要求

1. 工具安装源必须可控，不能直接随意访问公网。
2. 需支持离线包安装。
3. 需记录安装动作、版本、安装时间。
4. 缺失工具自动安装前需支持策略：
   - 自动安装
   - 需审批安装
   - 仅告警不安装
5. 需支持工具版本兼容矩阵。

# 9.8 日志采集、对比与分析模块

## 9.8.1 采集时机

1. 接入成功后初始采集
2. 每条用例执行前
3. 每条用例执行后
4. 失败时即时补采
5. 测试计划结束后汇总采集

## 9.8.2 采集范围

1. dmesg
2. journalctl / messages
3. RAID 事件日志
4. SMART / NVMe 健康日志
5. NIC 统计与 error counters
6. PCIe 错误信息
7. NPU / GPU 日志
8. BMC SEL / 传感器事件
9. 内核 panic / oops / mce / ras 记录
10. 自定义业务日志

## 9.8.3 分析能力

1. 前后对比：新增错误、计数变化、设备状态变化
2. 模式匹配：已知错误特征、关键告警关键字、版本相关已知问题
3. 聚类归并：相同根因的问题自动聚合
4. 异常分级：致命 / 高风险 / 一般 / 观察项
5. 关联分析：
   - 错误与执行步骤关联
   - 错误与配置变更关联
   - 错误与固件/驱动版本关联
6. 生成摘要：
   - 发生时间
   - 首次出现步骤
   - 涉及模块
   - 关键日志片段
   - 影响范围
   - 初步建议

## 9.8.4 行业补充要求

1. 平台不能只保存“失败结论”，还必须保存“失败上下文”。
2. 日志分析结果必须保留原始证据链接。
3. 应支持规则库和 AI 分析双轨输出：
   - 规则结论适合确定性问题。
   - AI 适合总结和辅助归因。
4. 高价值错误需支持沉淀为“问题知识卡片”，供后续复用。

# 9.9 AI 分析与推荐模块

## 9.9.1 AI 在平台中的定位

AI 是“分析助手”和“推荐引擎”，不是不可解释的最终裁决者。

## 9.9.2 AI 能力范围

1. **用例推荐**
   - 根据机器配置、历史问题、变更记录推荐测试集。
2. **日志总结**
   - 从大量日志中提取关键错误、时间线、模块归属。
3. **差异解释**
   - 对 BMC vs OS、BOM vs 实机的差异做自然语言解释。
4. **异常归因建议**
   - 给出可能根因和建议排查路径。
5. **报告自动撰写**
   - 自动生成结论、风险项、建议项。
6. **缺陷模板填充**
   - 自动整理复现条件、影响范围、关键日志、版本信息。
7. **知识库问答**
   - 面向内部工程师，回答“这类错误以前出现过吗”“哪类用例最适合当前机器”等。

## 9.9.3 AI 输出要求

1. 所有 AI 输出必须带来源：
   - 日志文件
   - 命令输出
   - 规则命中
   - 历史案例
2. AI 结论必须标注置信度。
3. AI 输出必须支持人工修订。
4. AI 不得直接覆盖原始测试结果，只能生成建议或辅助解释。

## 9.9.4 建议采用的 AI 方式

1. **RAG（检索增强生成）**：基于平台已有历史问题、版本说明、规则库进行回答。
2. **规则 + 模型混合引擎**：
   - 确定性检查使用规则。
   - 总结、聚类、归因使用模型。
3. **模板化生成**：
   - 报告、bug、总结使用结构化字段驱动生成，降低幻觉风险。

## 9.9.5 AI 风险控制

1. 避免 AI 编造不存在的硬件信息或日志内容。
2. 高风险场景（固件升级建议、判定硬件故障）需人工确认。
3. 建立 AI 输出审计记录。
4. 对用户敏感信息、凭据、日志脱敏后再供模型使用。

## 18. 主要风险与规避建议

### 风险 1：厂商工具碎片化

规避：插件化 + 适配层 + 命令能力注册表。

### 风险 2：AI 结论不可信

规避：规则引擎优先，AI 只做总结与建议；输出带证据引用。

### 风险 3：远程执行风险高

规避：命令白名单、高危动作审批、分环境隔离、最小权限。

### 风险 4：日志量过大

规避：对象存储 + 索引分离 + 分层归档。

### 风险 5：前端整机图维护成本高

规避：采用“模板化机型视图 + 动态模块映射”，不要每台机器手工画。
