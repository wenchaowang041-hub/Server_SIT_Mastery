# 《服务器测试百日学习计划——Day20：前20天系统总复盘，开始建立整机系统图》

大家好，我是 JACK，本篇是服务器测试百日学习计划 Day20。

到了 Day20，这一阶段不再继续补某个单独命令，也不再单独讲一个小模块，而是做第一次真正意义上的阶段验收。

因为前19天我们已经陆续学了：

- CPU 和 NUMA
- PCIe
- 存储
- 网络
- BMC
- BIOS / 固件
- 日志与驱动
- 自动化脚本

如果这些内容一直散着放，它们只是知识点；只有把它们串起来，才会变成整机视角。

所以 Day20 的主题很明确：

**前20天系统总复盘。**

这一天的目标，不是再学新东西，而是开始建立一张属于自己的整机系统图。

## 一、前20天到底在学什么

如果只看表面，前20天像是在学很多命令：

- `lscpu`
- `numactl -H`
- `lspci`
- `lsblk`
- `fio`
- `iperf3`
- `dmesg`
- `ipmitool`

但如果只把这些理解成命令集合，那就太浅了。

前20天真正建立的，不只是服务器各模块的基础知识，而是：

- 服务器的分层认知
- 整机结构理解
- 系统化测试思维
- 分层排障思维

也就是说，这20天不是在背命令，而是在搭一套“怎么看服务器”的脑图。

## 二、为什么 Day20 要开始画整机系统图

很多人学习服务器测试，到一个阶段就会卡住。

原因不是不努力，也不是命令不会，而是脑子里的内容太散。

比如：

- 知道 `lspci` 能看设备
- 知道 `numactl -H` 能看 NUMA
- 知道 `dmesg` 能看日志
- 知道 `ipmitool sel list` 能看历史事件

但一旦机器真的出问题，就不知道这些信息之间怎么连。

所以 Day20 的关键动作，不是再多查几个指标，而是把这些东西连起来，形成一张整机系统图。

这张图不是为了好看，而是为了以后遇到问题时，你脑子里能自动出现层次：

```text
CPU / NUMA
-> PCIe
-> 关键设备
-> OS / Driver
-> BMC / 日志 / 工具
```

有了这张图，很多问题才不会只停留在“现象层”。

## 三、整机系统图到底该怎么画

我建议第一版不要画得太复杂，先按 5 层来理解。

### 1. CPU / NUMA 层

这一层解决的是资源底座。

你要看：

- CPU 架构和型号
- Socket、Core、Thread、逻辑 CPU 数
- NUMA 节点数
- 每个 NUMA 节点的 CPU 范围
- 内存分布和 distance

这一层最重要的结论是：

**总 CPU 和总内存只能说明资源有多少，NUMA 才决定资源彼此离得近不近。**

### 2. PCIe 层

这一层解决的是设备挂载和链路问题。

你要看：

- 设备是否枚举
- 拓扑路径是否合理
- 速率是否正常
- 宽度是否正常
- 有没有降速、降宽

很多人一看到设备异常就先怀疑驱动，其实先看 PCIe 层经常更稳，因为设备能不能被系统使用，首先要建立在链路和枚举正常的前提上。

### 3. 关键设备层

这一层主要是：

- NVMe
- NIC
- NPU
- RAID
- HBA
- GPU（如有）

这些设备不是普通配件，而是决定整机能力的关键高速部件。

这一层的重点不是“设备很多”，而是：

- 存储能力从哪里来
- 网络能力从哪里来
- AI 加速能力从哪里来

### 4. OS / Driver 层

这一层必须单独拉出来，因为：

**设备在位，不等于设备可用。**

设备在 `lspci` 里能看到，只说明 PCIe 枚举层已经识别到它，不代表驱动已经加载成功，不代表功能已经正常，不代表性能已经达标。

所以这层要继续看：

- OS 是否识别设备
- 驱动是否正常加载
- 初始化是否成功
- 日志中是否有 error、reset、timeout

### 5. BMC / 日志 / 工具层

这一层属于观察层和验证层。

它不属于底层结构本身，而是工程师拿来取证和判断的入口。

这里要结合：

- BMC 页面
- 传感器
- SEL
- `dmesg`
- `journalctl`
- 测试工具结果

日志提供证据，BMC 提供带外视角，工具提供验证手段。  
但工具本身也可能带来偏差，所以工具输出不能直接等于最终结论。

## 四、为什么 Day20 很重要

Day20 最大的价值，是让你从“会查模块”往“会看整机”走。

过去你可能是这样：

- CPU 有 CPU 的命令
- 存储有存储的命令
- 网络有网络的命令
- BMC 有 BMC 的命令

但到了 Day20，你要开始把它们串成一条完整链：

```text
底层资源
-> 总线与拓扑
-> 关键设备
-> 系统识别
-> 日志与证据
-> 测试结果
```

这也是为什么 Day20 在计划表里被定义成“阶段验收”，而不是普通学习日。

因为从这一天开始，你看的就不再只是“一个命令有啥输出”，而是“这台服务器的结构和问题应该怎么组织起来”。

## 五、前20天最重要的10条结论

我把这20天最值得留下来的结论压成下面 10 条。

1. CPU 逻辑总数和内存总量只能说明服务器的资源上限，NUMA 和 PCIe 才决定真实的数据路径成本。  
2. `lspci` 看到设备，只能说明 PCIe 枚举层已经识别到设备，不代表设备功能已经完全正常。  
3. 设备是否真正可用，还必须继续结合 OS、驱动、日志和测试结果一起判断。  
4. 测试不能只看结果，还要结合日志、设计文档和现象做证据分析，再决定是否提交 bug。  
5. 遇到设备不识别或功能异常时，要按工具层、OS/驱动层、设备层、PCIe 层、物理连接层分层排查。  
6. 做测试前要先冻结配置和日志基线，测试后第一时间收集日志并保持环境一致，避免问题复现失真。  
7. 对简单重复的检查动作，应该沉淀成 SOP 或自动化脚本，减少时间浪费和人为手法误差。  
8. BMC、SEL、`dmesg`、`journalctl` 是整机测试中的关键证据源，不能只盯着测试工具输出。  
9. 存储、网络、NPU 这些关键设备的问题，最终都要回到 CPU、NUMA、PCIe 和驱动这条底层链路上理解。  
10. 前20天最重要的收获不是多会几个命令，而是开始具备“看到服务器能画出结构、分层定位问题”的基础能力。  

## 六、Day20 之后意味着什么

如果说 Day1 到 Day19 更多是在搭砖，那 Day20 就是在第一次把这些砖砌成墙。

从这一天开始，你应该逐渐形成这种能力：

- 看到一台机器，脑子里先出现结构
- 看到一个问题，脑子里先出现分层
- 看到一个异常，先找证据链，再下结论

这就是服务器测试从“会执行”往“会理解整机”走的分水岭。

## 七、最后总结

把 Day20 压成最短版，其实就是这 4 句话：

1. 前20天学的不是一堆命令，而是一套整机分层认知。  
2. 服务器应该按 CPU / NUMA、PCIe、设备、OS / Driver、BMC / 日志 / 工具这 5 层来理解。  
3. 能把这 5 层串起来，才算开始具备整机测试和整机排障的基础能力。  
4. Day20 的真正成果，不是学会新命令，而是开始建立属于自己的整机系统图。  

如果你现在也在学服务器测试，我建议 Day20 这种“阶段总复盘”一定不要跳过。因为它决定了你后面是在继续堆命令，还是开始真正理解整机。
