Day15 我们建立了一个很关键的意识：

**不是所有问题都叫硬件问题。**

那 Day16 就要继续往前走一步，解决另一个更关键的问题：

**当你怀疑驱动、怀疑系统、怀疑设备时，到底靠什么证明？**

答案就是：

**日志和驱动信息。**

排障不能只靠感觉，不能只说“我怀疑是驱动问题”，而是要把“怀疑”变成“证据”。

---

## 一、为什么 Day16 很重要

现场排障时，最常见的问题不是不会猜，而是猜得太快。

很多人看到现象就会直接说：

- 像驱动问题
- 像系统问题
- 像硬件问题

但真正稳定的工程化排障，不是“猜一个最像的原因”，而是：

```text
现象
-> 驱动状态
-> 内核日志
-> 时间线证据
-> 结论
```

Day16 的核心，就是把你从“猜问题”带到“拿证据”。

---

## 二、今天最重要的三个命令

Day16 的核心命令非常集中：

```bash
dmesg
lsmod
journalctl -k
```

不要小看这三个，它们基本能覆盖你第一轮排障里最关键的证据来源。

---

## 三、`lsmod`、`dmesg`、`journalctl -k` 分别看什么

### 1. `lsmod`

作用：

**看当前内核模块有没有加载。**

适合判断：

- 某个驱动模块是否已经在内核里
- 模块之间的依赖关系

如果设备 `lspci` 能看到，但 `lsmod` 里没有对应驱动模块，那第一反应就应该是：

- 驱动没装
- 驱动没加载
- 驱动和当前 kernel 不匹配

### 2. `dmesg`

作用：

**看内核启动和设备初始化过程。**

适合判断：

- 驱动有没有绑定成功
- 设备初始化是否正常
- 有没有 error / reset / timeout
- 有没有训练失败或异常恢复

### 3. `journalctl -k`

作用：

**看内核日志的时间线。**

适合判断：

- 启动时是正常还是异常
- 后续有没有 reset、掉线、报错
- 某个设备问题是启动就有，还是运行一段时间后才发生

> 一句话记忆：  
> `lsmod` 看模块，`dmesg` 看初始化，`journalctl -k` 看时间线。

## 四、为什么 `lspci` 能看到，还不够

这是 Day16 最典型的排障场景。

例如你现场看到：

- `lspci` 能看到设备
- 但工具认不到

很多人这时候会直接愣住，因为设备“明明在”。

但这里要建立一个非常重要的认知：

**PCIe 层可见，不等于驱动层正常；驱动层正常，也不等于工具层一定正常。**

也就是说，设备能出现在 `lspci` 里，只能说明：

- PCIe 总线已经枚举到了它

但还不能说明：

- 驱动一定加载了
- 驱动一定初始化成功了
- 上层工具链一定能识别它

这就是为什么 Day16 要补进来。

---

## 五、第一轮标准排障顺序

以后你看到“设备识别异常、工具认不到、行为不正常”这类问题，建议固定按这个顺序：

### 第1步：先确认设备在不在

```bash
lspci
```

如果 `lspci` 都看不到，那优先怀疑：

- 插槽
- 接线
- 拓扑
- 板卡
- 供电
- FW / BIOS / BMC / CPLD

### 第2步：再确认驱动在不在

```bash
lsmod
```

如果 `lspci` 能看到、`lsmod` 没有对应模块，第一判断就应该偏向：

- 驱动未安装
- 驱动未加载
- 驱动和 kernel 不匹配

### 第3步：再看初始化日志

```bash
dmesg | grep -i <关键字>
```

比如：

```bash
dmesg | grep -i nvme
dmesg | grep -i mlx
dmesg | grep -i hns
dmesg | grep -i pcie
dmesg | grep -i error
```

### 第4步：按时间线补看

```bash
journalctl -k
```

尤其适合看：

- 启动时正常，后面才出问题
- 设备反复 reset
- 某段时间出现大量错误

## 六、一个典型场景

假设现在出现下面这个现象：

- `lspci` 能看到设备
- 工具认不到
- `lsmod` 没有对应驱动模块
- `SEL` 没异常
- `sensor` 正常

那第一判断应该偏向哪一层？

正确方向应该是：

**优先判断为驱动层或软件适配问题，不优先判断为硬件健康问题。**

因为：

- PCIe 层已经看到设备
- BMC 健康层没有明显异常
- 驱动层却没有起来

这时重点应该是继续查：

- 驱动是否安装
- 驱动版本是否匹配
- 当前 kernel 是否兼容
- 日志里有没有加载失败

而不是一上来就说卡坏了。

---

## 七、为什么 Day16 本质上是在教“证据型排障”

你以后要慢慢把下面这种说法升级掉：

> 我感觉像驱动问题。

升级成：

> PCIe 层可见，但驱动模块未加载；日志里没有看到对应驱动绑定成功记录，因此当前更偏驱动或兼容性问题，而不是硬件本体故障。

两句话的区别很大：

- 前者是感觉
- 后者是证据

这就是 Day16 的价值。

## 八、今天适合输出什么

Day16 最适合输出：

- 日志摘录
- 驱动检查表
- 初步解释

例如：

| 来源 | 关键字 | 现象 | 初步解释 |
|---|---|---|---|
| `lsmod` | `mlx5_core` | 不存在 | 驱动未加载 |
| `dmesg` | `nvme` | 初始化成功/失败 | NVMe 驱动与设备关系 |
| `journalctl -k` | `pcie` | reset / error | PCIe 层异常 |

这种表格很适合后面做复盘和面试整理。

## 九、Day15 和 Day16 是怎么衔接的

Day15 解决的是：

**先判断问题属于哪一层。**

Day16 解决的是：

**再用日志和驱动信息去证明它属于哪一层。**

你可以这样记：

```text
Day15：先分层
Day16：再取证
```

这两天连起来，才是真正的工程化排障基础。

1. **排障不能只靠猜，要靠驱动和日志证据。**
2. **`lspci` 能看到设备，不代表驱动和工具层一定正常。**
3. **`lsmod` 看模块，`dmesg` 看初始化，`journalctl -k` 看时间线。**

## 十一、Day16 总结

Day16 的重点不是多背几个日志命令，而是建立“证据型排障”的习惯。

从今天开始，你要努力把自己的排障表达从：

- “我怀疑”

升级成：

- “日志里可以证明”

这一步一旦形成，后面无论是 NVMe、网卡、NPU、PCIe 还是整机 SIT 问题，你的判断都会更稳。
