## 一、白牌交换机不是“便宜交换机”，而是软硬件解耦平台

很多人第一次听“白牌交换机”，容易把它理解成低配版品牌机，这个理解不对。

白牌交换机（White-box Switch）本质上是：

- 硬件开放
- 软件可选
- 更适合大规模数据中心和 ODM 模式

和传统品牌交换机相比，最大差异不是“贵还是便宜”，而是**软硬件是否绑定**。

品牌交换机通常是：

- 原厂硬件
- 原厂封闭 OS
- 功能和责任边界清晰

而白牌交换机通常是：

- ODM 设计制造硬件
- 客户或方案方选择 NOS
- 平台、系统、模块、对端兼容都要做更多验证

对白牌交换机 SIT 来说，这意味着测试对象不再只是一个封闭盒子，而是一整条链路：

- ASIC SDK
- 平台驱动
- NOS 系统
- 光模块 / DAC / AOC
- EEPROM / BMC / 风扇 / PSU
- 对端服务器网卡和交换机

所以白牌交换机 SIT 的核心价值，不只是功能验证，而是：

- 平台 bring-up
- 兼容性验证
- 升级与回归
- 稳定性与异常恢复
- 问题定位和闭环

## 二、交换机的核心不是 CPU，而是 ASIC

服务器里，CPU 是绝对核心；交换机里，真正负责高速转发的是 ASIC。

ASIC（Application Specific Integrated Circuit）可以理解成交换机专用转发芯片，负责：

- 二层转发
- 三层查表
- ACL / QoS 等硬件能力
- Tbps 级别的数据面处理

这里要建立一个很关键的认知：

> 交换机的数据面主要走 ASIC，不走 CPU；CPU 更多负责控制面和管理面。

这也是服务器 SIT 和交换机 SIT 思维方式不同的地方。

服务器更强调：

- OS
- 驱动
- 应用路径
- 主机资源瓶颈

交换机更强调：

- 控制面
- 数据面
- 硬件表项
- 线速转发

常见交换芯片平台，面试里经常会提到：

- Broadcom Trident：偏均衡型，常见于企业网和中高密度场景
- Broadcom Tomahawk：偏高带宽、数据中心转发
- Broadcom Jericho：偏深缓存和大型汇聚
- Centec（盛科）：国产交换芯片代表

这里不一定要背得很细，但至少要知道平台定位有差异。

## 三、从服务器装机类比 ONIE：白牌交换机的安装入口

ONIE 全称是 Open Network Install Environment。

如果站在服务器工程师视角去理解，它很像：

- BIOS
- 加 PXE
- 再加一个最小化安装环境

它的任务很直接：

1. 把设备引导起来
2. 找到 NOS 镜像
3. 安装系统，比如 SONiC

常见安装方式包括：

- 网络安装：DHCP + HTTP/TFTP 拉镜像
- U 盘安装：现场单机调试常见
- BMC KVM 挂载安装：远程部署常见

如果你以前做过服务器装机、带外挂载、远程安装，这块迁移其实很自然。

从 SIT 视角，ONIE 相关测试重点一般包括：

- 镜像能否正常安装
- 安装失败是否有明确报错
- 异常断电后能否恢复
- 升级 / 回退 / 多镜像切换是否正常
- 批量部署一致性是否稳定

## 四、SONiC 才是白牌交换机里最该先搞懂的系统

SONiC（Software for Open Networking in the Cloud）是微软开源的交换机网络操作系统，底层基于 Debian Linux。

但如果只把 SONiC 理解成“一个 Linux”，那就太浅了。

SONiC 真正关键的三个点是：

- 容器化
- Redis 状态解耦
- 通过 SAI 屏蔽不同 ASIC 差异

### 1. SONiC 的几个关键组件

你至少要先记住这些：

- `database`：Redis 数据库，配置和状态中心
- `swss`：处理交换配置逻辑
- `syncd`：把配置同步到底层 ASIC
- `bgp`：运行 FRRouting 协议
- `teamd`：管理 LAG / LACP
- `lldpd`：负责邻居发现

### 2. 最关键的一条链路

```text
用户配置
-> Redis
-> swss
-> syncd
-> SAI
-> ASIC
```

这条链路理解透了，你对 SONiC 就不再只是“会点命令”的层面了。

### 3. SAI 是什么

SAI（Switch Abstraction Interface）是交换芯片抽象接口。

它的作用是把不同 ASIC 厂商的私有差异屏蔽起来，让 SONiC 上层逻辑不直接绑死在某一个 SDK 上。

所以一个比较像面试答案的说法是：

> SONiC 的关键不只是 Linux，而是容器化控制面、Redis 状态驱动，以及通过 SAI 统一不同 ASIC 平台。

### 4. SONiC 常用命令

初学阶段最值得先熟悉的是这些命令：

```bash
show interfaces status
show interfaces counters
show interfaces transceiver info Ethernet0
show interfaces transceiver dom Ethernet0
show vlan brief
show mac
show arp
show ip route
show bgp summary
show interfaces portchannel
show version
show logging
sudo config save -y
```

如果是做第一轮排障，最先要拿到的信息通常是：

- 接口状态
- 光模块和 DOM 数据
- VLAN / MAC / ARP
- PortChannel 状态
- 路由 / BGP 状态
- 日志输出

## 五、VLAN / Trunk / LAG：交换机二层最基础但最常考的三件事

### 1. VLAN

VLAN（Virtual Local Area Network）就是逻辑广播域隔离。

简单说：

- 同一个物理交换机上，可以划分多个二层网络
- 同 VLAN 默认二层互通
- 不同 VLAN 默认二层隔离

这和服务器侧的“不同网段要不要互通”很像，只不过这里先发生在交换机二层。

### 2. Access 口和 Trunk 口

Access 口一般接终端，只承载一个 VLAN。  
Trunk 口一般接交换机或虚拟化宿主机，可以承载多个 VLAN。

这里有一个非常容易答错的点：

> Trunk 不等于跨 VLAN 通信。

Trunk 只是把多个 VLAN 的流量透传过去，跨 VLAN 通信必须依赖三层转发。

### 3. LAG / LACP

LAG（Link Aggregation Group）是链路聚合，把多条物理链路捆绑成一条逻辑链路。

它主要有两个目的：

- 带宽叠加
- 链路冗余

LACP 是标准动态协商协议，聚合链路两端都需要正确配置。

从 SIT 视角，LAG 的重点不是“配起来就算完”，而是要看：

- 聚合是否成功
- 成员口状态是否一致
- 单链路故障时业务是否切换
- 恢复时间是否符合预期
- 哈希分流是否合理

## 六、为什么交换机性能测试总提 RFC2544

RFC2544 是网络设备性能测试的经典标准，最常见四项包括：

- 吞吐量 Throughput
- 丢包率 Frame Loss Rate
- 延迟 Latency
- 背靠背 Back-to-back

### 1. 吞吐量

吞吐量表示在零丢包前提下，交换机可以持续转发的最大速率。

对 100G 口来说，理想情况下应接近线速。

### 2. 丢包率

丢包率表示发出去的帧和收回来的帧之间的比例差值。

线速下低丢包甚至零丢包，通常是交换机的关键指标之一。

### 3. 延迟

延迟表示报文进入交换机后，到从出口发出的时间差。

交换机优劣不只是看能不能跑满，还要看时延是否稳定。

### 4. 背靠背

背靠背测试更偏缓存能力验证，适合模拟突发流量。

### 5. 为什么 64 字节小包最严苛

因为同样带宽下，小包数量最多，最考验 ASIC 的 PPS 处理能力。

所以交换机性能测试里，64B 小包成绩往往特别关键。

## 七、Redfish：交换机平台管理里和服务器最像的一块

很多服务器 SIT 工程师第一次看交换机，会以为全是网络协议；其实只要设备带 BMC，平台管理思路和服务器就有很强的相似性。

Redfish 是基于 HTTP/HTTPS 和 JSON 的带外管理接口，常见操作包括：

- 查询系统信息
- 查询温度、风扇和电源状态
- 配置 BMC 网口
- 执行开关机、重启等操作
- 固件升级

常见方法：

- `GET`：查询
- `POST`：执行动作
- `PATCH`：修改配置
- `DELETE`：删除资源

对于服务器 SIT 转交换机 SIT 的人来说，这一块反而是优势项：

> BMC、带外管理、Redfish、传感器健康监控，本来就是服务器和交换机平台之间共通的能力。

## 八、交换机 SIT 用例怎么设计

这部分和服务器 SIT 的方法论最像。

一个完整 SIT 用例，至少要包含四要素：

- 前置条件
- 测试步骤
- 预期结果
- 判定标准

而且不能只测 Happy Path，至少要覆盖三类场景：

- 正常场景
- 异常场景
- 边界场景

举几个简单例子：

- 正常场景：同 VLAN 两台主机互通
- 异常场景：不同 VLAN 默认不通；拔掉一颗 PSU 验证冗余
- 边界场景：创建最小/最大 VLAN ID；MAC 表写满后观察行为

真正拉开水平差距的地方，是**预期结果必须量化**。

不要写：

- “切换正常”
- “链路恢复快”
- “光功率正常”

要写成：

- LAG 单链路故障后，业务恢复时间 `< 500ms`
- ping 100 包，丢包率 `0%`
- DOM 读取的 TX / RX 功率落在规格书范围内

## 九、交换机排障怎么做：从下往上，一层层排

交换机问题最怕一上来就凭经验乱猜。

更稳妥的方法是从下往上逐层定位。

例如最常见的 “ping 不通”，可以按下面顺序查：

### 1. 物理层

- 接口状态是否 up
- 光模块是否识别正常
- DOM 光功率是否异常
- 线缆、光纤、模块是否有问题

常看命令：

```bash
show interfaces status
show interfaces transceiver info Ethernet0
show interfaces transceiver dom Ethernet0
```

### 2. 二层

- VLAN 是否配对
- MAC 是否学到
- 是否有错误包或丢包

常看命令：

```bash
show vlan brief
show mac
show interfaces counters
```

### 3. 三层

- 是否有目标网段路由
- ARP 是否正确
- 网关是否可达

常看命令：

```bash
show arp
show ip route
```

最后一个非常重要的原则是：

> 每次只改一个变量，不要同时换线、换模块、换口、换版本，不然问题现场很容易被自己破坏掉。

## 十、服务器 SIT 转白牌交换机 SIT，真正能迁移的能力是什么

这是我自己最关心的问题，也是很多转岗人最容易卡住的地方。

我的理解是：

**服务器 SIT 转交换机 SIT，不是从零开始，而是把已有的系统测试方法论迁移到新的测试对象上。**

可以直接迁移的能力包括：

- bring-up 和版本矩阵意识
- 兼容性测试思路
- 长稳、重启、异常场景验证
- BMC / Redfish / 传感器健康监控
- 日志保留和分层排障
- 自动化采集和回归思维

需要补齐的内容主要是：

- SONiC 架构
- SAI / ASIC 这一层认知
- VLAN / LAG / 路由基础
- 交换机性能测试标准
- 光模块和数据中心网络场景

换句话说，服务器 SIT 工程师不是不会做交换机 SIT，而是需要把：

“整机平台 + 管理面 + 稳定性”

扩展成：

“交换平台 + 控制面 + 数据面 + 网络场景”

## 十一、写在最后

白牌交换机 SIT 的知识体系看起来跨度很大，但如果本身做过服务器 SIT、整机测试、兼容性验证或平台联调，真正需要补的并没有想象中那么夸张。

最重要的不是一开始就把所有协议背全，而是先建立这几个认知：

- 白牌交换机的本质是软硬件解耦
- ONIE 是安装入口，SONiC 是主流 NOS
- SONiC 的关键是容器化、Redis 和 SAI
- 交换机数据面转发主要靠 ASIC
- SIT 工作重点在平台、兼容性、升级回归、性能和排障

先把这条主线搭起来，再继续往上补：

- BGP
- EVPN / VXLAN
- MLAG
- RoCE
- PFC / ECN

这时候学习路径就会顺很多。

如果你也是从服务器 SIT 往交换机 SIT 方向准备，这篇文章可以先当成一份入门底稿，后面再逐步扩展成自己的知识体系。
