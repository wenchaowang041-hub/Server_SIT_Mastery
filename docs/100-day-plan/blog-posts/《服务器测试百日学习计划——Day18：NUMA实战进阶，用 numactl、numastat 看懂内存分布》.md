## 一、为什么 Day18 要做实战

NUMA 不是背概念就够了，真正做服务器测试时，你最常遇到的是这些问题：

- 机器总内存很多，但某个节点已经快满了
- CPU 看起来没满，程序却一直跑不快
- 网卡、NVMe、NPU 都挂在某个节点，但业务线程跑到了另一个节点
- 同一台机器，换了绑核方式，性能差距很大

这些问题的共同点是：

**你必须先知道“资源在哪个节点上”，再谈“性能为什么这样”。**

所以 Day18 重点不是再讲一次 NUMA 定义，而是把它落到命令、输出和判断方法上。

## 二、先把机器的 NUMA 结构看清楚

最先用的还是这两个命令：

```bash
lscpu | grep NUMA
numactl -H
```

### `lscpu` 看总览

`lscpu` 更适合先看全局视角，比如：

- 机器有几个 socket
- 有几个 NUMA 节点
- 每个节点大概对应多少 CPU

### `numactl -H` 看细节

`numactl -H` 更关键，因为它能直接告诉你：

- 有几个节点
- 每个节点有哪些 CPU
- 每个节点有多少内存
- 节点之间的距离是多少

你可以把它理解成：

- `lscpu` 是“总览图”
- `numactl -H` 是“节点地图”

如果你后面要做网卡、NVMe、NPU、IRQ、业务线程分析，这张图必须先看明白。

---

## 三、`numactl -H` 里最该看的 4 件事

### 1. 节点数量

先确认机器到底是单 NUMA 还是多 NUMA。

单 NUMA 不代表没有问题，但多 NUMA 机器里，节点之间的距离和归属一定要重点看。

### 2. 每个节点的 CPU 范围

这一步的意义是确认：

- 某个节点上的 CPU 是否集中
- 业务线程有没有被绑到更合适的节点
- 中断有没有集中打到少数 CPU 上

### 3. 每个节点的内存大小

这个信息能帮你判断：

- 节点容量是否均衡
- 有没有某个节点明显更小
- 机器是不是存在插条不均衡、BIOS 配置异常或者硬件缺失

### 4. distance

`distance` 是 NUMA 实战里最容易被忽略，但也最重要的字段之一。

你可以简单记成：

- 同节点访问最便宜
- 近节点次之
- 远节点最贵

一旦业务线程、内存和设备不在同一条近路上，延迟和吞吐就可能开始变差。

---

## 四、`numastat`：看内存是不是“落在了对的地方”

如果说 `numactl -H` 是看结构，那 `numastat` 更偏向看**实际内存使用情况**。

常用命令：

```bash
numastat
numastat -p <pid>
```

### `numastat` 看整机

它能帮你粗略判断：

- 哪个节点的内存压力更大
- 有没有明显偏向某个节点
- 是否存在跨节点内存分配的迹象

### `numastat -p <pid>` 看进程

这个更实用。你可以直接看某个进程的内存分布，判断：

- 这个进程主要吃的是哪个节点的内存
- 它是不是已经跑偏到别的节点去了
- 绑定 CPU 之后，内存有没有跟着一起落到同节点

这一步在排查：

- 大内存应用
- AI 推理
- 数据库
- 高吞吐网络服务

时都很常见。

---

## 五、`/sys` 目录：定位设备归属最直接

如果你想知道某个 PCIe 设备到底挂在哪个 NUMA 节点，最直接的办法还是看 `sysfs`。

例如：

```bash
cat /sys/bus/pci/devices/0000:35:00.0/numa_node
cat /sys/bus/pci/devices/0000:81:00.0/numa_node
```

这个值的作用非常直接：

- `0 / 1 / 2 / 3` 之类的数字，表示设备归属的 NUMA 节点
- `-1` 往往表示当前系统没能明确识别，或者设备归属信息不可用

这个字段在 Day4 其实已经碰过了。Day18 只是把它从“看设备”推进到“看业务路径”。

比如：

- 网卡在 node1
- NVMe 在 node1
- 业务线程也跑在 node1

这通常比：

- 网卡在 node1
- 业务线程在 node3
- 内存却主要从 node0 分配

要合理得多。

## 六、实战判断：什么情况下要怀疑跨 NUMA

下面这些情况，基本都该先看 NUMA：

- 吞吐不差，但延迟抖动大
- CPU 没满，但程序跑不快
- 网卡速率正常，业务转发却不稳定
- NVMe 顺序性能看起来还行，但 4K 随机和高并发明显掉
- AI 任务在多卡场景下性能不一致

判断顺序建议是：

```text
先看设备在哪个 NUMA
再看线程在哪个 NUMA
最后看内存有没有跟着走
```

如果三者没有对齐，问题就可能不是“设备坏了”，而是“路径不顺”。

---

## 七、两个最常见的误区

### 误区1：只看 CPU 数量

CPU 多，不代表路径好。

真正关键的是：

- CPU 在哪个节点
- 内存在哪个节点
- 设备挂在哪个节点
- 业务有没有跑到同一个节点

### 误区2：只看总内存

总内存大，不代表每个节点都健康。

你要看的不是“整机有没有 1TB”，而是：

- node0 有多少
- node1 有多少
- node2 有多少
- node3 有多少

这才是 NUMA 视角。
