**中断与网卡性能。**

```text
IRQ -> CPU -> NUMA -> 网卡性能
```

---

## 一、为什么 Day13 要学 IRQ

很多人对网络性能的理解停留在下面这几层：

- 网口 `UP`
- `ethtool` 显示速率正常
- `iperf3` 能跑

但这些只说明“链路建立了、业务流量能过去”，不代表性能路径是最优的。

真正影响吞吐和延迟的，除了链路本身，还包括：

- 网卡中断打到了哪些 CPU
- 这些 CPU 是否集中在少数几个核上
- 这些 CPU 和网卡所在 NUMA 是否靠近
- 业务线程是否又跑在另一个 NUMA 节点上

也就是说，**链路能通，不代表路径合理。**

---

## 二、先把 IRQ 理解清楚

IRQ（Interrupt Request）可以先简单理解成：

**设备向 CPU 发出的“我这边有事，你来处理一下”的通知。**

对网卡来说，常见触发中断的场景包括：

- 收到数据包
- 发送完成
- 队列事件
- 错误状态变化

CPU 收到中断后，才会继续处理后续协议栈路径和业务数据。

所以你现在要建立一个新认知：

**网卡性能不只是网卡自己的事，它和 CPU 的处理中断路径直接相关。**

---

## 三、为什么 IRQ 会影响网卡性能

如果中断分布不合理，常见后果包括：

- 少数 CPU 被打得很忙
- 其他 CPU 明明空着，但没参与处理
- 包处理不及时，排队变长
- 软中断压力变大
- 吞吐下降
- 延迟和抖动上升
- 跨 NUMA 访问增多

最典型的坏路径是这种：

```text
网卡挂在 NUMA1
IRQ 主要落在 NUMA0
业务线程跑在 NUMA3
```

这就意味着数据路径、中断处理路径、业务处理路径没有对齐。

结果往往不是“完全不能用”，而是：

- 能跑
- 但跑不满
- 而且不稳定

---

## 四、链路层问题和处理路径问题要分开看

网络性能差，常见有两类问题：

### 第一类：链路层问题

比如：

- 协商速率不对
- 双工异常
- 光模块不兼容
- 网线或交换机端口问题
- PCIe 链路降速降宽

这类问题优先看：

```bash
ethtool eno1
ethtool -i eno1
lspci -vv
```

### 第二类：处理路径问题

比如：

- IRQ 集中在少数 CPU
- IRQ 和网卡 NUMA 不匹配
- 业务线程和设备 NUMA 不一致
- CPU 热点明显

这类问题优先看：

```bash
cat /proc/interrupts
lscpu
numactl -H
```

> **判断顺序要固定：先看链路，再看路径。**

---

## 五、核心命令：`/proc/interrupts`

今天最重要的文件就是：

```bash
cat /proc/interrupts
```

它的作用是把系统当前各类中断统计直接摊开给你看。

一般你会看到：

- 每一列对应一个 CPU
- 每一行对应一种中断源
- 最后通常带设备或驱动名称

比如网卡相关中断，经常能在行尾看到：

- 网口名
- 驱动名
- 队列名

常见过滤方法：

```bash
grep eno /proc/interrupts
grep -i eth /proc/interrupts
grep -i mlx /proc/interrupts
grep -i hns /proc/interrupts
```

不同平台名字不一样，所以不要死盯一个关键字。

1. 网卡相关中断有哪些
2. 它们主要落在哪些 CPU
3. 这些 CPU 和设备所在 NUMA 是否大致对齐

这一步走通，Day13 就已经学到了最关键的 80%。

## 七、标准分析流程

### 第1步：先确认你看的到底是哪张网卡

```bash
ip link
ethtool -i eno1
```

重点记录：

- 网口名
- 驱动名
- bus-info

因为后面要把：

- 网口
- PCIe 地址
- IRQ
- NUMA

这些东西串起来看。

### 第2步：找出网卡中断

```bash
cat /proc/interrupts
grep eno /proc/interrupts
grep -i hns /proc/interrupts
```

你不是只要看到“有中断”就结束，而是要注意：

- 是哪几行
- 对应哪些队列
- 计数主要集中在哪些 CPU 上

### 第3步：判断是否过度集中

如果一张网卡的大部分中断只集中在 `CPU0`、`CPU1` 这类极少数核上，你第一反应就该担心：

- 会不会造成热点 CPU
- 会不会导致队列排队
- 会不会让吞吐上不去
- 会不会引发抖动和延迟增加

这时问题往往不是“网卡坏了”，而是“活都堆到少数 CPU 上了”。

### 第4步：和 NUMA 做一次对照

```bash
lscpu
numactl -H
```

你要回答的问题是：

- 这些 CPU 属于哪个 NUMA
- 网卡大概率挂在哪个 NUMA
- 中断落点和设备 NUMA 是否靠近

如果设备在 `NUMA1`，但 IRQ 主要落在 `NUMA0`，业务线程又跑在 `NUMA3`，那么路径就会很分散，远程访问和跨节点开销都会增加。

---

## 八、一个典型思考题

假设：

- 网卡挂在 `NUMA1`
- IRQ 落在 `NUMA0`
- 业务线程跑在 `NUMA3`

那会发生什么？

结论通常是：

- 中断处理不在设备最近的一侧
- 业务处理又在另一侧
- 路径被拉长
- 跨 NUMA 甚至跨 socket 开销增加
- 延迟变大，抖动变大，吞吐下降

如果 `NUMA0` 和 `NUMA1` 还在同一个 socket 上，影响可能相对小一些；如果已经跨 socket，性能损失通常更明显。

## 十一、今天最容易犯的错

### 错误1：只看链路，不看中断

链路 `UP` 只代表链路建立，不代表性能路径已经健康。

### 错误2：只看有没有中断，不看分布

真正影响性能的，不是“有没有中断”，而是中断落在哪些 CPU 上。

### 错误3：只看 CPU 忙不忙，不看 NUMA

CPU 没打满，不代表路径是优的。跨 NUMA 一样可能拖垮吞吐和延迟。

### 错误4：一看到吞吐低就只怪交换机或网线

链路层当然要查，但处理路径问题同样常见，而且更容易被忽略。

---

## 十二、今天必须背下来的 3 句话

1. **网口 `UP` 不代表性能路径健康。**
2. **网络性能问题要分链路层和处理路径两类看。**
3. **处理路径重点就是：网卡 NUMA -> IRQ CPU -> 业务线程 CPU 是否对齐。**
