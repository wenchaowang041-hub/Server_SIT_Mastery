**第一篇 从零认识服务器**

*信创服务器的组成、分类与核心特点*

**一、什么是服务器**

服务器是专门为其他设备提供服务的高性能计算机。你每天用的微信、刷的抖音、点的外卖，背后都是一台台服务器在支撑。

服务器与普通PC的本质区别不是性能，而是设计目标：

- **稳定性**：7×24小时不间断运行，PC可以随时关机重启，服务器不行

- **可靠性**：关键部件冗余设计（双电源、多路网卡），单点故障不影响整体

- **可管理性**：支持远程管理、监控、固件升级，无需到机房现场

- **可扩展性**：CPU、内存、硬盘、网卡灵活扩展

**二、服务器的外观与规格**

企业级服务器通常安装在标准机柜中，外形扁平横置，高度用 U 来衡量（1U = 4.4cm）。标准机柜高度42U，能安装42台1U服务器。

| **规格** | **高度** | **适用场景**                        |
|----------|----------|-------------------------------------|
| 1U       | 4.4 cm   | 高密度部署，Web/应用服务器          |
| 2U       | 8.8 cm   | 主流规格，存储和计算均衡            |
| 4U       | 17.6 cm  | 高配，大量硬盘扩展                  |
| 6U       | 26.4 cm  | 特殊场景，如我们测试的NPU加速服务器 |

**三、服务器的六大核心部件**

**▍ ① CPU — 计算大脑**

服务器通常配备2块CPU（双路），我们测试的服务器使用**华为鲲鹏920**，ARM架构，单颗64核，双路共128核。

**▍ ② 内存 — 临时存储**

服务器内存支持**ECC（错误纠正码）**，能自动检测并纠正数据错误。我们这台装了512GB DDR4 ECC。

**▍ ③ 存储 — 持久化数据**

包含系统盘和数据盘。主流方案是NVMe SSD（通过PCIe直连CPU），读写速度可达3000MB/s以上。

**▍ ④ 网卡 — 网络通信**

服务器网卡带宽远高于PC，常见25GbE、100GbE。通常做Bond（链路聚合），多口绑定提升带宽和冗余。

**▍ ⑤ 电源 — 稳定供电**

双电源冗余设计，一块故障时另一块立即接管，支持热插拔，不需要关机更换。

**▍ ⑥ BMC — 远程管理**

BMC（基板管理控制器）是服务器的小管家，即使主系统关机也在独立运行，可以远程开关机、查温度、查日志。

**四、服务器的三种形态**

| **类型**        | **特点**                               | **适用场景**       |
|-----------------|----------------------------------------|--------------------|
| 机架式（Rack）  | 扁平横置，插入标准机柜，最常见         | 数据中心大规模部署 |
| 塔式（Tower）   | 外形像立起来的PC机箱，不需要机柜       | 小企业、办公室     |
| 刀片式（Blade） | 多台刀片插入刀箱共享电源散热，密度最高 | 超大规模数据中心   |

**五、信创服务器**

信创（信息技术应用创新）服务器采用国产核心技术：国产CPU（鲲鹏920、飞腾、龙芯）+ 国产OS（openEuler、麒麟、统信UOS）+ 国产配套软件。

我们当前测试的就是典型信创服务器：**华为鲲鹏920 CPU + 昇腾Atlas 300I NPU加速卡**，运行openEuler和Klas系统。

> *💡 鲲鹏920是SoC集成设计（CPU + 南桥 + 网卡Hi1822 + SAS控制器集成在一起），与x86模块化设计不同，网络和存储验证必须和CPU一起测。*

**二、鲲鹏920架构特点**

- **ARM架构**：基于ARMv8.2指令集，与x86的Intel/AMD完全不同的指令集

- **SoC集成设计**：CPU + 南桥 + 网卡(Hi1822) + SAS控制器集成，不是x86的模块化架构

- **高核心数**：单颗最多64核，双路128核，适合高并发场景

- **无超线程**：每核1线程，Thread(s) per core = 1，不像Intel的2线程

> *💡 鲲鹏920的SoC集成设计是信创测试的重点：网络和存储性能必须和CPU一起测，不能像x86平台那样模块化单独测。*

**三、NUMA架构**

鲲鹏920双路服务器有4个NUMA节点，每个节点包含32个CPU核心和本地内存。访问本地内存（NUMA距离=10）比跨Socket访问（距离=32-33）快很多。

>
> node 0 1 2 3
>
> 0: 10 16 32 33
>
> 1: 16 10 33 32
>

>

>
> *💡 NPU加速卡和NVMe硬盘的NUMA归属对AI推理性能影响很大，尽量让NPU、NVMe和CPU在同一NUMA节点，避免跨NUMA传输。*

**一、内存是什么**

内存（RAM，Random Access Memory）是服务器运行时临时存放数据的地方，CPU处理任务时所需的数据都先加载到内存里，速度极快，但断电后数据消失。

服务器内存和普通PC内存相比：

- 容量更大，通常从64GB起步，高配可达TB级别

- 支持 **ECC（错误纠正码）** 功能，自动检测并纠正内存数据错误，保障数据准确性

- 支持多通道，提升内存带宽

- 更注重稳定性和长时间运行可靠性

重点关注 **Triad 综合带宽**，如果跑分明显偏低，检查：

- 内存通道是否插满

- 内存是否按推荐槽位对称安装

- 内存频率是否正确配置

**▍ 2. 压力测试**

**memtest** — 专业内存检测工具，逐一检测每个内存单元是否存在故障：

> \# 安装memtest
>
> yum install memtest86+
>
> \# 运行memtest（通常在开机引导阶段运行）
>
> memtest86+

memtest 会对内存进行多轮读写校验，发现错误会直接报告出错的内存地址，适合深度检测内存硬件问题。

**stress/stress-ng** — 对内存施加持续压力，验证稳定性：

> \# stress对内存加压，占用80%内存跑1小时
>
> stress --vm 4 --vm-bytes 80% --timeout 3600
>
> \# stress-ng更精细的内存压力模式
>
> stress-ng --vm 4 --vm-bytes 80% --vm-method all --timeout 3600

压测期间同步监控内存使用情况：

> \[root@localhost ~\]# watch -n 2 free -h
>
> total used free shared buff/cache available
>
> Mem: 503Gi 398Gi 92Gi 234Mi 12Gi 102Gi
>
> Swap: 0B 0B 0B
>
> \# 确认内存压力真正跑上去

**五、常见问题及排查**

**▍ 内存压力告警导致测试中断**

这是内存压测中最常见的问题，压力跑到一半系统触发告警甚至中断测试。

**1. 内存温度过高**

- 查看 ipmitool 温度传感器，确认内存温度是否超过阈值

- 正常内存工作温度一般不超过 85℃

- 如果温度异常高，检查机房环境温度和服务器风道是否畅通

**2. 内存出现硬件错误**

查看系统日志有无内存报错：

> \[root@localhost ~\]# dmesg \| grep -i memory
>
> \[ 123.456789\] EDAC MC0: 1 CE memory read error on CPU_SrcID#0_MC#0_Chan#0_DIMM#0
>
> \# CE表示可纠正错误，UE表示不可纠正错误，UE需要立即更换内存条

**3. 内存容量不足**

压测占用内存过大导致系统 OOM（内存溢出），适当调低压测内存占用比例：

> \# 改为占用60%内存
>
> stress --vm 4 --vm-bytes 60% --timeout 3600

**4. ECC错误触发**

服务器 ECC 机制检测到内存错误自动告警，查看 BMC Web 端的系统事件日志（SEL），也可以用命令查看：

> \[root@localhost ~\]# ipmitool sel list
>
> 1 \| 03/02/2025 \| Memory \| Correctable ECC \| Asserted
>
> \# 出现ECC告警需记录并评估是否需要更换内存

**六、测试时需要记录的数据**

| **指标**     | **说明**                        |
|--------------|---------------------------------|
| 总内存容量   | 是否和规格一致                  |
| 内存频率     | 是否达到标称值                  |
| STREAM Triad | 内存综合带宽跑分                |
| 内存最高温度 | 压测期间峰值温度，正常不超过85℃ |
| ECC错误次数  | 压测期间是否有ECC告警           |
| 压测时长     | 建议至少跑2小时以上             |

**七、总结**

内存测试核心关注三点：**读写速率达标、压力稳定、无ECC报错**。遇到压力告警中断，优先查温度和系统日志，定位是温度问题还是硬件错误。

**第四篇 存储详解：NVMe/U.2/M.2接口与fio测试**

*NVMe · SATA · fio · smartctl*

大家好，我是JACK，本篇是服务器硬件基础系列第四篇，上一篇我们讲了内存，这篇我们聊存储，结合实际测试经验来讲。

**一、存储是什么**

存储负责持久化保存数据，断电后数据不丢失。和内存不同，存储的特点是容量大、数据永久保存，但速度相对内存慢很多。

服务器常见存储类型：

- **HDD（机械硬盘）**：容量大、价格低，但速度慢，适合海量冷数据存储

- **SSD（固态硬盘）**：无机械结构，速度远超HDD，稳定性好，适合系统盘和热数据

- **NVMe SSD**：通过PCIe接口直连CPU，速度是SATA SSD的好几倍，是企业级服务器高性能存储的主流选择

**二、存储接口类型详解**

服务器存储接口种类较多，不同接口对应不同的使用场景，了解区别对测试和选型都很重要。

| **接口类型** | **协议**      | **速度**     | **形态**    | **适用场景**          |
|--------------|---------------|--------------|-------------|-----------------------|
| SATA         | SATA          | 最高6Gbps    | 2.5寸/3.5寸 | 普通SSD/HDD           |
| SAS          | SAS           | 最高22.5Gbps | 2.5寸/3.5寸 | 企业级HDD/SSD         |
| M.2          | SATA或NVMe    | 最高32Gbps   | 小型插卡    | 消费级/轻薄服务器     |
| U.2          | NVMe          | 最高32Gbps   | 2.5寸       | 企业级NVMe SSD        |
| U.3          | NVMe/SAS/SATA | 最高32Gbps   | 2.5寸       | 新一代企业级，兼容U.2 |
| PCIe AIC     | NVMe          | 最高128Gbps  | PCIe插卡    | 极高性能场景          |

重点说一下 **M.2 和 U.2 的区别**：

**U.2：**

- 2.5寸标准尺寸，企业级设计

- 支持热插拔，硬盘故障可以不关机直接更换

- 散热好，适合长时间高负载运行

- 通过标准接口连接NVMe背板，背板通过PCIe总线与CPU通信

- 我们测试的服务器NVMe硬盘就是U.2接口，通过NVMe背板连接

**M.2：**

- 体积小，消费级产品为主

- 不支持热插拔

- 散热相对较差，长时间高负载容易温度过高

- 企业服务器上使用时通常需要转接板，将M.2转换成PCIe接口插到主板PCIe插槽上，通过PCIe总线与CPU通信

- 我们测试的服务器M.2硬盘就是通过转接板转成PCIe接口使用

**四、存储测试方法**

存储测试主要验证顺序读写和随机读写性能，使用 fio 工具测试。

**▍ 1. 顺序读写测试**

顺序读写反映大文件连续读写的速度，适合评估备份、日志等场景性能：

> \# 顺序写测试
>
> \[root@localhost ~\]# fio --name=seq_write --filename=/dev/nvme0n1 \\
>
> --rw=write --bs=1M --size=10G --numjobs=1 --iodepth=32 \\
>

**一、网卡是什么**

网卡（NIC，Network Interface Card）负责服务器的网络通信，是服务器连接外部网络的核心硬件。

服务器网卡和普通PC网卡相比：

- 带宽更高，常见10GbE、25GbE、100GbE

- 支持多端口，一块网卡可以有2个、4个甚至更多网口

- 支持Bond（链路聚合），多个网口合并提升带宽和冗余

- 部分高端网卡支持RDMA，大幅降低网络延迟

- 更注重长时间高负载下的稳定性

**三、Bond链路聚合**

Bond是将多个网口绑定在一起的技术，可以提升带宽和冗余。服务器测试中常用的有Bond1和Bond4两种模式。

**Bond1（Active-Backup，主备模式）：**

- 同一时间只有一个网口在工作，另一个备用

- 主网口故障时自动切换到备用网口

- 带宽不叠加，仍然是单口速率

- 主要用途是保障网络冗余，防止单点故障

**Bond4（802.3ad LACP，链路聚合模式）：**

- 多个网口同时工作，带宽叠加

- 比如两个25GbE口做Bond4，理论带宽达到50GbE

- 同时也有冗余保障，一个口故障不影响整体

- 需要交换机也支持LACP协议配合

简单对比：

| **模式** | **带宽** | **冗余** | **适用场景**               |
|----------|----------|----------|----------------------------|
| Bond1    | 不叠加   | ✅       | 对冗余要求高，带宽要求不高 |
| Bond4    | 叠加     | ✅       | 既要高带宽又要冗余         |

**一、电源是什么**

电源（PSU，Power Supply Unit）负责为服务器所有模块提供稳定的电力供应。服务器电源和普通PC电源相比，更强调稳定性、冗余性和高效率。

服务器电源的核心特点：

- **冗余设计**：通常配备2块或以上电源，一块故障时另一块立即接管，服务器不中断运行

- **热插拔**：故障电源可以在不关机的情况下直接拔出更换

- **高效率**：企业级电源通常有80 PLUS认证，能效转换率更高

- **远程监控**：通过BMC实时监控电源状态、功耗、电压等参数

**一、散热是什么**

服务器运行时CPU、内存、NVMe硬盘、NPU加速卡等部件会持续产生热量，散热系统的作用就是把这些热量及时排走，确保各部件在安全温度范围内稳定运行。

散热不足的后果：

- 温度触发告警，BMC降低风扇转速限制已经不够用时反而触发降频

- 长期高温加速硬件老化，缩短使用寿命

- 极端情况下触发热保护自动关机，测试中断

**二、风冷架构**

我们测试的鲲鹏920服务器采用风冷散热，6U机箱，风扇分两组：

- **前置风扇**：5个，负责机箱前段进风，主要覆盖内存、存储区域

- **中置风扇**：8个，负责机箱中段主散热，覆盖CPU、NPU加速卡等核心部件

气流方向：前进后出，冷空气从机箱前端进入，经过各部件吸热后从机箱后端排出。

> *💡 ：服务器必须安装在标准机柜中，前后进出风通畅。裸机放桌上测试时要注意风道，不能堵住进出风口，否则散热效率大幅下降，温度数据也不具备参考价值。*

**一、BMC是什么**

BMC（Baseboard Management Controller，基板管理控制器）是服务器主板上独立运行的管理芯片，有自己的CPU、内存和网络接口，即使服务器主系统关机，BMC依然在运行。

我们测试的鲲鹏920服务器使用的是**华为自研BMC**，底层基于ASPEED芯片，上层运行华为自研的管理固件，通过独立的管理网口对外提供服务。

BMC的核心能力：

- **远程控制**：远程开机、关机、重启，无需到机房现场操作

- **硬件监控**：实时监控温度、风扇转速、电压、功耗等传感器数据

- **日志记录**：记录系统事件日志（SEL），保存硬件层面所有重要事件

- **远程KVM**：通过Web界面远程查看服务器屏幕输出，相当于远程接了一套键鼠显示器

- **固件管理**：远程升级BIOS、BMC固件

**二、IPMI协议**

IPMI（Intelligent Platform Management Interface）是BMC对外提供的标准管理接口协议，ipmitool 就是通过IPMI协议与BMC通信的命令行工具。

> \# 本地调用（在服务器上直接运行）
>
> ipmitool \<命令\>
>
> \# 远程调用（从其他机器管理目标服务器）
>
> ipmitool -I lanplus -H \<BMC_IP\> -U \<用户名\> -P \<密码\> \<命令\>

| **指标**    | **说明**                   |
|-------------|----------------------------|
| BMC固件版本 | ipmitool mc info确认       |
| SEL日志     | 测试前清空，测试后保存归档 |
| 传感器基线  | 空载状态快照，压测对比用   |
| 峰值功耗    | dcmi power reading最大值   |

**八、总结**

BMC是服务器测试中最重要的管理工具，贯穿测试全流程：**测试前清空日志、测试中监控传感器、测试后保存归档**。华为自研BMC在IPMI标准接口基础上扩展了 dcmi power reading 等功耗相关命令，结合 ipmitool sensor list 可以全面覆盖温度、风扇、电源等所有监控需求。

**第九篇 BIOS/UEFI详解：鲲鹏920关键配置项与测试注意事项**

*华为自研UEFI · NUMA/PCIe/CPU功耗/内存频率*

大家好，我是JACK，本篇是服务器硬件基础系列第九篇，也是本系列的最后一篇，上一篇我们讲了BMC/IPMI，这篇我们聊BIOS/UEFI，结合实际测试经验来讲。

**一、BIOS/UEFI是什么**

BIOS（Basic Input/Output System）是服务器上电后最先运行的固件，负责初始化所有硬件，然后将控制权移交给操作系统。

现代服务器已经全面升级为 **UEFI**（Unified Extensible Firmware Interface），相比传统BIOS：

- 支持2TB以上磁盘引导

- 图形化操作界面，支持鼠标

- 启动速度更快

- 支持安全启动（Secure Boot）

- 可扩展性更强

日常我们仍然习惯叫它"BIOS"，本篇统一使用这个叫法。

鲲鹏920服务器使用的是**华为自研BIOS**，基于UEFI标准，在ARM架构下针对鲲鹏920平台做了专项适配和优化。

**二、如何进入BIOS**

服务器开机POST阶段按指定键进入BIOS，鲲鹏920服务器通常是 **Delete** 键或 **F2** 键。

实际测试中更常用的方式是通过**BMC Web远程KVM**进入BIOS，不需要到机房现场操作：

1\. 浏览器登录BMC Web界面

2\. 进入远程控制 → KVM

3\. 服务器重启时在KVM画面中按对应键进入BIOS

> *💡 ：POST阶段时间很短，要在出现华为Logo时迅速按键，错过了就要再重启一次。用BMC KVM操作时网络延迟会影响按键响应，可以提前把键盘输入焦点切到KVM窗口，重启后连续按键。*

**三、鲲鹏920 BIOS关键配置项**

**▍ 1. NUMA配置**

NUMA（Non-Uniform Memory Access）是影响服务器性能的重要架构特性，鲲鹏920双路服务器有4个NUMA节点。

> 路径：Advanced → NUMA Configuration

| **选项** | **说明** |
|----|----|
| NUMA Support | Enable（默认开启，不要关闭） |
| Node Interleaving | Disable（默认关闭，开启后内存访问均匀化但单NUMA性能下降） |

> *💡 ：保持默认配置，即NUMA Support开启、Node Interleaving关闭，这样可以测出最真实的NUMA性能表现。如果应用层需要关闭NUMA感知，在OS层用 \`numactl\` 处理，不建议在BIOS层关闭。*

**▍ 2. 内存频率配置**

> 路径：Advanced → Memory Configuration

| **选项**         | **说明**                                              |
|------------------|-------------------------------------------------------|
| Memory Frequency | 建议设为Auto或最高支持频率                            |
| Memory RAS Mode  | Independent Channel（性能优先）/ Mirror（可靠性优先） |

内存频率如果没有跑到标称值，先到BIOS确认这里的配置是否正确，有时候设置为了节能模式导致频率降低。

**▍ 3. PCIe配置**

> 路径：Advanced → PCIe Configuration

| **选项**          | **说明**                                               |
|-------------------|--------------------------------------------------------|
| PCIe Speed        | Auto（建议）/ Gen4 / Gen3                              |
| PCIe ASPM         | Disable（测试时建议关闭，避免省电策略影响PCIe性能）    |
| Above 4G Decoding | Enable（支持大于4GB的PCIe内存映射，NPU加速卡必须开启） |

> *💡 ：之前遇到过PCIe链路跑不到Gen4速率，lspci看LnkSta只有8GT/s而不是16GT/s，进BIOS检查发现PCIe Speed被设置成了Gen3，改成Auto后恢复Gen4正常速率。*

**▍ 4. CPU功耗配置**

> 路径：Advanced → CPU Configuration → Power Management

| **选项** | **说明** |
|----|----|
| CPU Power Management | Performance（测试时建议选性能模式） |
| CPU Frequency Scaling | Disable（测试时建议关闭变频，保持固定频率） |
| C-States | Disable（测试时建议关闭，避免CPU进入低功耗状态影响响应延迟） |

> *💡 ：性能测试时关闭C-States和变频，确保CPU始终以最高频率运行，测试结果更稳定。功耗测试时可以开启，测试省电效果。*

**▍ 5. 安全启动（Secure Boot）**

> 路径：Security → Secure Boot

信创环境通常需要安装国产OS（如openEuler、麒麟），如果遇到OS无法启动的情况，检查Secure Boot是否开启，安装非签名系统时需要临时关闭。

> Secure Boot：Disabled

**▍ 6. 启动顺序配置**

> 路径：Boot → Boot Option Priorities

可以设置从哪个设备优先启动：NVMe硬盘、SAS硬盘、PXE网络启动等。测试时如果需要从外部介质重装系统，需要临时调整启动顺序。

**五、常见问题及排查**

**▍ 1. 服务器开机卡在POST阶段**

通过BMC KVM查看屏幕输出，观察卡在哪一步：

- **卡在内存检测**：可能有内存条故障或接触不良，重新插拔或更换内存条

- **卡在PCIe设备初始化**：某个PCIe设备有问题，检查lspci识别情况

- **卡在RAID初始化**：RAID卡配置有问题，进BIOS或RAID卡管理界面检查

**▍ 2. 安装OS后无法引导**

> \# 常见原因和处理：
>
> 1\. Secure Boot开启 → 进BIOS关闭Secure Boot
>
> 2\. 启动顺序错误 → 进BIOS调整Boot Option到正确的硬盘
>
> 3\. 引导分区损坏 → 需要重装系统或修复引导

**▍ 3. PCIe设备识别不全**

已在Day4文章中详细说明，BIOS层面重点检查：

- Above 4G Decoding 是否开启

- PCIe Speed配置是否正确

- 是否需要升级BIOS版本

> \# 确认系统层识别情况
>
> \[root@localhost ~\]# lspci \| grep -i ascend
>
> \# 对比实际安装数量，如果不一致进BIOS确认设备枚举情况

**▍ 4. 内存频率跑不到标称值**

> \# 先查看当前实际运行频率
>
> \[root@localhost ~\]# dmidecode -t memory \| grep "Configured Memory Speed"
>
> Configured Memory Speed: 2933 MT/s
>
> \# 标称3200 MT/s，实际只跑2933
>
> \# 进BIOS检查
>
> \# Advanced → Memory Configuration → Memory Frequency
>
> \# 如果是Auto但仍然降频，检查内存混插情况（不同频率内存混插会降到最低公共频率）

**六、BIOS升级**

BIOS升级通常在以下情况下进行：

- 修复已知硬件兼容性问题

- 新增对某型号硬盘/网卡/加速卡的支持

- 修复安全漏洞

升级方式：

**方式一：通过BMC Web界面升级（推荐）**

1\. 登录BMC Web界面

2\. 进入固件升级页面

3\. 上传华为提供的BIOS固件包

4\. 确认升级，等待完成后自动重启

**方式二：通过BIOS界面本地升级**

1\. 将固件包放到U盘

2\. 进入BIOS → Firmware Update

3\. 选择U盘中的固件包升级

> *💡 ：BIOS升级过程中不能断电，建议升级前备份当前BIOS配置，升级后重新核对关键配置项是否被重置为默认值。*

**七、总结**

BIOS是服务器硬件测试中经常需要介入的环节。鲲鹏920服务器BIOS测试关注四个核心配置：**NUMA开启、PCIe满速、CPU性能模式、内存频率达标**。遇到硬件识别异常、性能跑不到标称值，除了检查OS层配置，一定要同步检查BIOS配置是否正确。
