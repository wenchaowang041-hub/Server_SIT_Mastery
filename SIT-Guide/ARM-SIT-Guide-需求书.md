## 1. 项目目标

建设一个面向 ARM 架构服务器 SIT 测试工作的内部知识与执行门户，用于统一沉淀测试方法、测试用例、问题定位经验、自动化脚本规范和项目交付模板。

网站需要同时满足三类需求：

- 新人快速学习 ARM 服务器 SIT 测试体系
- 测试人员按模块执行测试并查找命令、步骤、判定标准
- 项目交付时快速生成测试计划、测试报告和问题定位记录

本项目不只是做一个资料展示页，而是要沉淀一套 ARM 服务器 SIT 测试工作台。用户打开网站后，应该能完成“查资料、找命令、跑用例、看标准、定位问题、生成交付材料”的完整链路。

### 1.1 建设原则

- 以测试执行为主线，不以资料堆叠为主线。
- 以故障现象和测试模块双入口组织内容。
- 命令、步骤、判定标准必须结构化。
- 所有内容都要能持续维护，避免一次性页面。
- 第一阶段不依赖后端，保证本地和内网环境可用。

### 1.2 不做范围

第一阶段暂不实现：

- 用户登录和权限管理
- 在线多人协作编辑
- 后端数据库
- 自动从设备拉取测试结果
- 与 Jira / 禅道 / GitLab 的深度集成
- 复杂报表系统

## 2.1 核心使用场景

### 场景一：新人学习

新人第一次接触 ARM 服务器测试，需要从首页进入学习路径，按顺序了解 ARM 架构、服务器硬件、启动链路、BMC、OS、网络、存储和稳定性测试。

验收要求：

- 首页提供“新人学习路径”入口。
- 每个模块有基础概念、常用命令、测试重点和常见问题。
- 关键术语有解释或跳转链接。

### 场景二：测试执行

测试工程师拿到一台 ARM 服务器，需要按测试阶段完成冒烟、功能、性能、稳定性和回归测试。

验收要求：

- 能按测试阶段筛选用例。
- 每个用例有前置条件、步骤、预期结果和判定标准。
- 命令可一键复制。
- Checklist 可本地勾选。

### 场景三：问题定位

测试过程中出现掉盘、网口不通、BMC 无响应、Kernel panic 等问题，需要快速找到排查流程。

验收要求：

- 问题定位手册支持按故障现象查找。
- 每类问题包含快速判断、排查命令、日志路径和升级研发所需材料。
- 能从测试模块跳转到相关故障页面。

### 场景四：项目交付

测试负责人需要整理测试计划、测试报告、风险项和遗留问题。

验收要求：

- 提供标准测试计划模板。
- 提供测试报告模板。
- 提供版本矩阵模板。
- 提供风险评估模板。
- 提供交付 Checklist。

## 3.1 信息架构设计

网站采用“三层入口”：

- 第一层：按测试领域进入，例如 BMC、网络、存储、稳定性。
- 第二层：按测试阶段进入，例如冒烟、功能、性能、回归、交付。
- 第三层：按问题现象进入，例如无法启动、掉盘、网口不通、BMC 异常。

这样可以同时满足学习、执行和排障三种使用方式。

## 5. 核心功能需求

### 5.1 文档导航

网站应提供左侧树形导航，支持按模块浏览：

- 知识类：ARM 基础、硬件架构、启动链路
- 测试类：BMC、网络、存储、性能、稳定性、兼容性
- 执行类：测试用例、Checklist、自动化脚本
- 交付类：测试计划、测试报告、问题单、风险评估
- 排障类：启动失败、掉盘、网口不通、BMC 异常、性能低

导航要求：

- 支持一级和二级分类。
- 当前页面高亮。
- 移动端折叠为抽屉导航。
- 页面刷新后仍能定位到当前模块。

### 5.2 全局搜索

搜索范围包括：

- 文档标题
- 测试项名称
- 命令关键字
- 错误日志关键字
- 故障现象
- 测试工具名称
- 硬件模块名称

搜索结果应展示：

- 标题
- 所属模块
- 摘要
- 标签
- 最后更新时间

搜索交互要求：

- 输入关键字后实时过滤。
- 搜索结果按相关度展示。
- 支持无结果提示。
- 点击结果后跳转到对应内容块。
- 支持命令关键字搜索，例如 `fio`、`iperf3`、`ipmitool`、`dmesg`。

### 5.5 Checklist 功能

测试文档应支持 Checklist：

- 前置条件已确认
- 固件版本已记录
- OS 版本已记录
- 测试工具已安装
- 测试步骤已执行
- 日志已保存
- 结果已判定
- 问题已登记

Checklist 至少应支持本地勾选保存。

Checklist 数据保存要求：

- 第一阶段使用浏览器 localStorage 保存。
- 勾选状态与页面 ID 绑定。
- 提供“重置当前 Checklist”按钮。
- 不要求多人同步。

### 5.6 测试用例库

测试用例库应支持表格展示：

| 字段 | 说明 |
| --- | --- |
| 用例编号 | 测试用例唯一编号 |
| 测试模块 | BMC / 网络 / 存储 / 性能等 |
| 测试项 | 具体测试对象 |
| 优先级 | P0 / P1 / P2 / P3 |
| 前置条件 | 执行前需要满足的条件 |
| 测试步骤 | 详细操作步骤 |
| 预期结果 | 正常表现 |
| 判定标准 | Pass / Fail 条件 |
| 关联脚本 | 自动化脚本路径 |
| 适用平台 | 适用服务器或板卡 |
| 维护人 | 用例负责人 |
| 更新时间 | 最近更新时间 |

用例优先级定义：

- P0：冒烟 / 阻塞级
- P1：核心功能
- P2：常规覆盖
- P3：扩展验证

测试用例库筛选条件：

- 模块
- 优先级
- 测试阶段
- 适用 OS
- 是否可自动化
- 关键字

测试用例详情应支持从表格展开或跳转到详情页。

### 5.7 问题定位手册

问题定位手册应按现象组织：

- 服务器无法上电
- 无法进入 BIOS / UEFI
- 无法进入 OS
- 网口不通
- NVMe 掉盘
- PCIe 设备丢失
- BMC Web 无法访问
- IPMI 命令无响应
- 性能低于预期
- 压力测试死机
- Kernel panic
- 风扇满转
- 传感器异常

每个问题页面应包含：

- 现象描述
- 影响范围
- 快速判断
- 排查流程
- 相关命令
- 日志路径
- 常见原因
- 解决方案
- 升级给研发所需材料

网站定位为工程测试工作台，视觉风格应克制、清晰、偏工具型。

## 14. 风险与约束

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 内容过多导致页面复杂 | 用户找不到重点 | 第一阶段只交付核心模块，其他模块简版或占位 |
| 测试用例格式不统一 | 后续难以搜索和生成报告 | 统一用例字段和编号规则 |
| 命令缺少适用条件 | 误用命令或误判结果 | 每组命令标注适用场景和注意事项 |
| 静态页面后期维护困难 | 内容增长后难维护 | 第二阶段拆分 JSON 和 Markdown |
| Checklist 只存在本地 | 不能团队共享 | 第一阶段接受，第三阶段再做后端同步 |
