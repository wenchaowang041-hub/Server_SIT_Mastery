ARM 基础知识

ARM 架构服务器与 x86 的关键差异，帮助测试人员理解 ARM 平台的测试特点。

ARMv8 / ARMv9 架构

特性说明
AArch64ARM 64 位执行状态，对应 x86_64
指令集RISC 架构，固定长度 32 位指令，Load/Store 模型
寄存器31 个通用寄存器 (X0-X30)，X30 为链接寄存器
内存模型弱内存模型 (Weak Memory Model)，需要显式内存屏障 (DMB/DSB/ISB)

异常级别 (Exception Levels)

级别用途对应 x86
EL0用户空间应用Ring 3
EL1Linux 内核Ring 0
EL2虚拟化 (Hypervisor)VMX Root
EL3安全监视器 (TF-A / ATF)SMM

GIC 中断控制器

ARM 使用 GIC (Generic Interrupt Controller) 管理中断，与 x86 的 APIC 不同。GIC 支持 SPI (共享外设中断)、PPI (私有外设中断)、SGI (软件生成中断)。测试中关注中断分配和亲和性。

SMMU / IOMMU

ARM 的 SMMU (System Memory Management Unit) 等效于 x86 的 IOMMU/VT-d，负责 DMA 地址转换和隔离。关闭 IOMMU 可降低延迟，但失去 DMA 保护。

NUMA 架构

Kunpeng 920 双路服务器有 2 个 NUMA 节点。跨 NUMA 访问增加 20-50% 延迟。程序应绑定到设备所在节点运行。

lscpu | grep NUMA
numactl --hardware
numactl --cpunodebind=0 --membind=0 ./program

Cache 架构

ARM 处理器通常有 L1 (指令/数据分离)、L2 (统一)、L3 (共享) 三级 Cache。Cache 一致性由硬件维护，但弱内存模型要求正确插入内存屏障。

ARM 与 x86 差异对测试的影响

差异测试影响
弱内存模型需要验证同步原语正确性，多线程测试更关注内存序
GIC vs APIC中断分配和亲和性配置方式不同
SMMU vs IOMMUDMA 地址转换和隔离机制不同
UEFI + Device TreeARM 平台可能使用 Device Tree 而非纯 ACPI
TF-A / ATF多了一层固件启动链路，调试需要考虑 EL3

固件与启动链路

ARM 平台启动链路：BootROM → TF-A → U-Boot → EDK2/UEFI → OS

启动链路

BootROM

CPU 内部固化的第一级启动代码，从 SPI Flash/eMMC 加载 BL1

TF-A (Trusted Firmware-A)EL3 安全固件，初始化安全世界，加载 BL2/BL31

U-Boot

第二级 bootloader，加载 UEFI 或直接加载 Kernel

EDK2 / UEFI

UEFI 固件，提供 ACPI / SMBIOS 表，引导 OS

ACPI / Device Tree

硬件描述表，ARM 平台可能使用 Device Tree 或 ACPI

Kernel

Linux 内核启动，加载驱动，挂载文件系统

固件管理

组件说明
Secure Boot验证启动链完整性，每个阶段签名验证
PXE网络启动，用于 OS 安装和自动化部署
ISO 安装通过 BMC 虚拟光驱挂载 ISO 安装系统
固件升级BIOS/BMC 固件升级，注意过渡版本

启动失败排查

阶段排查

无法进入 OS检查 UEFI boot order / 系统盘识别 / 内核启动参数
Kernel panic串口抓取 panic 日志，检查驱动/内存/ACPI 表

BMC 管理

BMC 是服务器的"第二操作系统"，即使 OS 崩溃也能远程管理硬件。

测试目的

验证 BMC 管理功能完整性，包括 Web 界面、IPMI、Redfish、SOL、传感器、FRU、风扇策略、电源控制和固件升级。

BMC Web 界面

BMC 提供 Web 管理界面，功能包括：

  • 传感器实时监控（温度/电压/风扇/功耗）
  • 风扇策略配置与状态查看
  • 电源控制（开/关机/重启/DC cycle/AC cycle）
  • 固件升级（BIOS/BMC/CPLD/NVMe/网卡）
  • 虚拟控制台 (KVM / HTML5)
  • SEL 事件日志查看与导出
  • FRU 信息查询
  • 用户管理与权限配置
  • NTP / SMTP / LDAP 配置

Redfish API

RESTful API 管理接口，替代传统 IPMI。S920X20 支持 iBMC Redfish 接口。

# 获取系统信息
curl -k -u admin:Admin@9000 https://<bmc_ip>/redfish/v1/Systems/1

# 获取传感器数据
curl -k -u admin:Admin@9000 https://<bmc_ip>/redfish/v1/Chassis/1/Thermal

# 获取电源信息
curl -k -u admin:Admin@9000 https://<bmc_ip>/redfish/v1/Chassis/1/Power

# 控制电源状态
curl -k -X PATCH -H "Content-Type: application/json" \
  -d '{"PowerState":"On"}' \
  -u admin:Admin@9000 https://<bmc_ip>/redfish/v1/Systems/1

# 获取 BMC 固件信息
curl -k -u admin:Admin@9000 https://<bmc_ip>/redfish/v1/UpdateService/FirmwareInventory

# 修改启动设备
curl -k -X PATCH -H "Content-Type: application/json" \
  -d '{"Boot":{"BootSourceOverrideTarget":"Pxe"}}' \
  -u admin:Admin@9000 https://<bmc_ip>/redfish/v1/Systems/1

SOL (Serial over LAN)

通过 IPMI 网络通道重定向串口，用于远程查看 BIOS 启动日志和 Kernel 串口输出。

# 配置 SOL
ipmitool -I lanplus -H <bmc_ip> -U admin -P password sol activate

# 设置 SOL 参数
ipmitool sol set volatile-bit-rate 115.2
ipmitool sol set non-volatile-bit-rate 115.2

# 退出 SOL（转义序列）
~.  (tilde + dot)

# 通过 ssh 连接 SOL
ssh -p 2200 admin@<bmc_ip>

异常处理方式
BMC Web 无法访问检查网络 → ping BMC IP → telnet 443 → 冷重启 BMC
IPMI 命令无响应ipmitool mc reset cold 冷重启 BMC

# BMC 冷重启（不影响 OS）
ipmitool mc reset cold

网络测试

测试目的

验证 ARM 服务器上所有网口识别正常、链路协商正确、吞吐率达到标称值、bonding/VLAN/RDMA 功能正常。

前置条件

  • 所有网口物理连接正常
  • 对端交换机/服务器已配置
  • iperf3 已安装（两端）
  • 已记录网卡 BDF 和驱动信息

测试步骤

网口识别

lspci | grep -i ethernet ip -br link

Link 检查

ethtool eth0 — 确认 Link detected: yes，Speed 正确

DHCP / 静态 IP

ip addr add 192.168.1.100/24 dev eth0

iperf3 吞吐量

iperf3 -c <server_ip> -P 8 -t 300

bonding

配置 bond0 (mode 4 LACP)，验证聚合带宽

VLAN

ip link add link eth0 name eth0.100 type vlan id 100

RDMA / RoCE

rdma link 查看 RoCE 绑定，测试 rping

中断亲和性

确认网卡中断分布在正确的 NUMA 节点 CPU 上

判定标准

测试项Pass 条件
1G 网口iperf3 ≥ 940 Mbits/sec
10G 网口iperf3 ≥ 9.4 Gbits/sec
25G 网口iperf3 ≥ 23 Gbits/sec (MTU=9000)
bonding聚合带宽 ≥ 单口带宽 × N × 0.9

日志采集

# 采集网络日志
ip -s link > net_ip_link.log
ethtool -S eth0 > net_ethtool_S.log
dmesg | grep -iE "hns3|mlx5|eth" > net_dmesg.log

Checklist

  • 所有网口已识别
    lspci / ip link 显示所有网口
  • Link 状态正常
    ethtool 确认 Link detected: yes
  • iperf3 带宽达标
    ≥ 标称值 90%
  • bonding 功能正常
    聚合带宽达标，拔线测试通过
  • VLAN 隔离正常
    不同 VLAN 无法互通
  • 日志已归档
    ethtool/dmesg/iperf3 日志已保存

存储测试

测试目的

验证 ARM 服务器上所有存储设备识别正常、SMART 健康、性能达标、热插拔可靠、RAID 配置正确、长稳无异常。

前置条件

  • 所有 NVMe / SATA / SAS 硬盘已安装
  • RAID 卡（如有）已安装并配置
  • fio / nvme-cli / smartmontools 已安装
  • 已记录固件版本和设备信息

测试步骤

磁盘识别

lsblk -d nvme list lspci | grep -i nvme lsscsi

NVMe SMART 检查

nvme smart-log /dev/nvme0 — 确认 critical_warning=0，温度正常，无介质错误

磁盘性能测试 (fio)

fio --name=randread --rw=randread --bs=4k --iodepth=32 --numjobs=4 --time_based --runtime=60 --direct=1 --ioengine=libaio

磁盘热插拔

运行时拔出 NVMe 再插入，确认重新识别，无数据丢失

RAID 配置验证

storcli64 /c0/show — 所有 VD Online，无 Degraded

文件系统测试

mkfs.ext4 格式化后挂载，读写验证

I/O 压力测试

fio 混合压力 12H+，确认无掉盘无报错

掉盘定位排查

记录 PCIe 链路、NVMe 驱动、SMART 状态

背板排查

通过背板连接的 NVMe，验证背板信号完整性

NUMA 绑定调优

ARM 双路服务器有 2 个 NUMA 节点，跨 NUMA 访问增加 20-50% 延迟。程序应绑定到设备所在节点运行。

# 查看 NUMA 拓扑

# 绑定到 NVMe 所在 NUMA 节点

# 绑定到网卡所在 NUMA 节点
numactl --cpunodebind=1 --membind=1 iperf3 -s

# 查看进程的 NUMA 分布
numastat -p $(pidof program)
numactl --show

最小化硬件

只保留 CPU + 1 DIMM + 电源，尝试上电

BMC SEL 日志 + 串口日志 + 最小化配置清单

无法进入 BIOS/UEFI → 排查流程

现象

开机后无法进入 BIOS 设置界面，或 UEFI 启动卡住。

排查流程

通过 SOL 查看启动日志

ipmitool sol activate

确认串口输出是否正常到 UEFI 阶段

清除 NVRAM

清除 BIOS 设置，恢复默认

升级 BIOS 固件

检查是否为固件 Bug

升级材料

串口完整启动日志 + BIOS 版本 + NVRAM 清除前后对比
